# Medic for Apache Spark - First Aid for Failing Jobs - Drasko Profirovic, Pinterest

## TL;DR
Drasko Profirovic, a staff engineer at Pinterest, presents "Medic," an agentic diagnostics tool for troubleshooting Apache Spark job failures. He walks through the evolution from a single ReAct agent prototype to a multi-agent architecture, detailing lessons learned around prompt engineering, log/metrics handling, testing, and observability that ultimately enabled reliable, scalable Spark failure analysis.

## Key Takeaways
- Supporting Spark at scale is hard: support rotations are relentless, priorities are ambiguous, and troubleshooting distributed systems is difficult, especially for newcomers.
- The vision for Medic was to ask "Why did a job fail?" and receive a deep research document with evidence-based root causes and suggested fixes, available on surfaces like Slack and Airflow.
- The initial single ReAct agent with one monolithic prompt was unsustainable—prompt tuning degraded other behaviors, response quality was inconsistent, and large log outputs quickly consumed context windows.
- Pinterest invested heavily in observability (OpenTelemetry traces to LangFuse) and testability (an end-to-end test harness with record/playback fixtures and offline evaluations) to quantify quality and prevent regressions.
- Logs are noisy and full of benign exceptions; an exception classifier pipeline fingerprints, clusters, ranks, and filters likely red herrings, exposing only the top K exceptions via MCP tools.
- Raw time-series metrics are token-inefficient; a quarantine sub-agent converts metrics into annotated graph collages (resembling Grafana dashboards) and uses vision to reason about patterns, keeping token usage bounded regardless of job duration.
- The system evolved into a multi-agent architecture built on LangGraph's deep agent library, with each agent having a dedicated prompt, a subset of MCP tools, and built-in tools like to-do lists and a virtual file system.
- The workflow includes intent classification, a triage agent generating failure hypotheses, parallel research agents scoring and validating hypotheses, a supervisor selecting the highest-confidence root cause, and a healer agent offering remediations from runbooks in a vector database.
- Multi-agent architecture provided the greatest control over system behavior and made expanding scope as simple as adding a new prompt—enabling extensions like Spark SQL optimization.
- Future directions include incorporating user feedback for automatic improvement and applying the same pattern to other distributed systems like Flink and Trino.

## Detailed Breakdown

### Motivation and Vision [00:01](https://www.youtube.com/watch?v=0RNNfxpdbQk&t=1s)
Drasko introduces himself as a staff engineer at Pinterest and outlines the talk's scope: why Medic was built, the journey from prototype to current architecture, lessons learned, and what's next. He reflects on his experience across multiple companies in data platform organizations, noting a common thread: a high bar for quality support to partner teams. Support rotations feel like a never-ending stream of problems, and troubleshooting Spark—or any distributed system—is inherently difficult, especially for newcomers. He highlights the challenge of ambiguous priorities—deciding whether to help one team with a failing job or unblock another with a looming deadline. Unlike humans, LLMs can scale knowledge and capabilities on demand. The vision for Medic was to simply ask "Why did a job fail?" and receive a deep research document with evidence on the root cause and suggested fixes grounded in the job's context, available on surfaces like Slack and the Airflow UI.

### Initial Prototype and Early Challenges [02:07](https://www.youtube.com/watch?v=0RNNfxpdbQk&t=127s)
The team began by exposing data resources via the Model Context Protocol (MCP) to connect them to LLMs. They could start an LLM conversation with MCP tools enabled and ask the model to reason about a Spark job. This worked but required careful prompting from the human operator. They extended the prototype with a single reasoning and acting (ReAct) agent, given one prompt embodying the problem-solving approach, response structure, and examples of common failure patterns. Beta trials revealed significant shortcomings: prompt tuning became unsustainable because one prompt had to do everything, and improving one area degraded another. Response quality was inconsistent—sometimes shallow, sometimes too verbose. They lacked controls to keep the agent on track, and large tool outputs from logs quickly consumed tokens, halting reasoning. Their end-to-end testing relied on manual tests from production, which felt anecdotal since production data was retentioned away, making it hard to know if changes broke earlier wins.

### Observability and Testability Investments [03:42](https://www.youtube.com/watch?v=0RNNfxpdbQk&t=222s)
To improve the system, the team invested in observability and testability. They used OpenTelemetry to publish traces to LangFuse, viewing the agent's execution as a waterfall diagram of steps to understand the causes of lower-quality responses. The reliance on manual testing highlighted the need for a scalable solution. They built an end-to-end test harness to snapshot production state and codify expectations as offline evaluations. In record mode, the agent calls real downstream systems and tool responses are captured as fixtures saved to the file system and checked in as code. In playback mode, the agent runs against fixtures instead of production data, performs the analysis, and generates a report. The test suite grades the report based on offline evals—for example, checking that the agent suggests no more than three fixes, scoring lower if too many are provided to manage verbosity. This allowed the team to quantify quality rather than rely on intuition and gave confidence that improvements didn't introduce regressions.

### Log Handling and Exception Classification [05:46](https://www.youtube.com/watch?v=0RNNfxpdbQk&t=346s)
Logs are noisy, and many exceptions are benign; simply focusing on the last exception is not always suitable. Initially, the team used a heuristics-based approach with regex to filter certain exceptions, but this didn't scale. They built an exception classifier pipeline that learns which exceptions commonly appear in successful jobs, treating those as likely red herrings and filtering them out. The agent fingerprints and clusters exceptions, then ranks them by content relevance and recency relative to job termination. Instead of consuming logs directly, the agent is given two MCP tools: get the top K truncated exceptions, or get full log details for a specific exception. This improved the signal-to-noise ratio and reduced the chance of the LLM anchoring its investigation on a misleading exception.

### Metrics Analysis via Vision [06:47](https://www.youtube.com/watch?v=0RNNfxpdbQk&t=407s)
Raw time-series metrics are not context-window friendly; feeding raw data to an LLM works at small scale but fails for long-running production jobs and is token-inefficient. The team performs metrics analysis in a quarantine sub-agent that converts raw time-series data into graphs, collaged into a final image resembling a Grafana dashboard with annotations like min and max values. The image is attached to the LLM conversation, and the model is prompted to reason about patterns. This approach guarantees a bounded number of input tokens regardless of job duration. Useful signals include executors dropping to zero or near zero, long plateaus or bottlenecks, and any resource behavior inconsistent with healthy progress. The sub-agent summarizes findings and returns results to the parent agent, keeping the context window healthy.

### Multi-Agent Architecture Overhaul [08:23](https://www.youtube.com/watch?v=0RNNfxpdbQk&t=503s)
The team overhauled the agent harness, moving from a single ReAct agent to a multi-agent architecture built on LangGraph's deep agent library. Each agent now has a dedicated prompt and a subset of MCP tools, while the deep agent library provides built-in tools like a to-do list and a virtual file system to keep agents on track—an approach mirroring coding tools like Claude Code or Codex. This refactor decomposed the single prompt into specialized roles, providing clear separation and making it easier for developers to maintain and test each prompt independently. A pleasant consequence was that expanding scope became as simple as adding a new prompt, which enabled extending Medic to help users optimize Spark SQL jobs.

### Workflow and Current Architecture [09:28](https://www.youtube.com/watch?v=0RNNfxpdbQk&t=568s)
The workflow begins with the user's request entering the system, where intent is classified as either requiring a simple answer or a deep diagnostic session. For the latter, a triage agent determines the Spark job's lifecycle state and, if failed, uses a subset of tools to generate failure hypotheses. Each hypothesis is researched in parallel by research agents that gather evidence, return a score, and identify a root cause. The supervisor selects the highest-confidence root cause and invokes a healer agent to offer remediations based on runbooks ingested into a vector database. Finally, the supervisor assembles the formatted report. The multi-agent architecture proved very effective, offering the greatest control over system behavior. Enhancements to log handling substantially reduced inaccurate root causes. The team trialed LangGraph's workflows for more determinism but found them brittle compared to the ReAct paradigm.

### Future Directions [10:31](https://www.youtube.com/watch?v=0RNNfxpdbQk&t=631s)
Current experiments include incorporating user feedback from prior sessions to automatically improve the agent. The team also sees a broader opportunity to apply the Medic pattern to other distributed systems like Flink and Trino. Drasko closes by acknowledging the contributors who made the project possible.
