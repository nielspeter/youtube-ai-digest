## TL;DR
Manoj Nair, Snyk's Chief Innovation Officer and CTO, argues that the fundamental architectural decision for agentic security is separating the generator from the validator — AI models cannot reliably serve as both. Drawing on real data from thousands of enterprise customers, he demonstrates that AI-generated code quality is worse than human code, agentic environments (skills, MCP servers) are being poisoned, and agents are creating ungoverned attack surfaces, all while vulnerability backlogs grow quarter over quarter.

## Key Takeaways
- **Generator-validator separation is the core architectural principle**: AI models that generate code or actions cannot be the same systems that validate them for security — probabilistic systems alone miss too much (only 50–75% issue detection rates versus deterministic checks).
- **Autonomous attacks are real and present**: Attackers using frontier lab models with good context and harnesses can chain low-severity vulnerabilities into exploits, eliminating the old strategy of only fixing "criticals and highs."
- **AI-generated code quality is measurably worse than human code**, compounding existing vulnerability backlogs — Snyk's 4,800+ customers saw 108% quarter-over-quarter backlog growth.
- **The agentic environment is a new, largely ungoverned threat surface**: Skills, MCP servers, and agent behaviors introduce risks like toxic skills (over a third contain malware or vulnerabilities), poisoned MCP servers, and agents autonomously creating unauthorized copies of PII data.
- **You can't govern what you don't know exists**: For every model found in a repository, there are on average three times more agentic components — the full landscape must be mapped.
- **Model performance varies dramatically by use case**: Recent Snyk red-teaming showed frontier models excelled at blocking PII extraction but a hot new open model allowed 100% PII extraction; conversely, that same open model resisted decision-override attacks 0% of the time.
- **Deterministic and probabilistic systems must work together**: LLMs should augment, not replace, deterministic security checks to cover surfaces that rules-based systems cannot reach.
- **Snyk's Evo platform and agentic dev security offering** (GA'd the day of the talk) address prevention, remediation, environment scanning, and real-time governance of agentic development.
- **AI governance cannot live in a Confluence page or PDF** — it must be enforced in real time within the loops where agents and developers operate.

### Cutting Through the AI Fog [01:35](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=95s)
Nair takes the stage, noting Snyk secures about 5,000 enterprise customers including half of the Fortune 500, and shares real data from those customers. He references the talk title "cutting through the AI fog" and explains that having security tracks like this is how the industry cuts through it — security must be part of the room without slowing things down. He introduces the central thesis: can the generator and the validator be the same? Snyk's position, supported by data, is that they cannot.

### Three Core Customer Problems [03:39](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=219s)
Nair frames the fundamental question: can you trust what your agents just shipped and how they did it? He identifies three problems he hears from customers: autonomous attacks, untrusted output environments and agent behavior, and ungoverned AI application building.

### Autonomous Attacks and Compounding Vulnerabilities [04:41](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=281s)
Autonomous attacks are not hypothetical — all frontier labs have been used in automated attacks, and it can be done even without frontier models. With good context and a good harness, attackers now have a system that never sleeps. This fundamentally disrupts application security: you can no longer practice contextual risk management by fixing only criticals and highs, because low vulnerabilities can be chained into exploits. Nair notes that a thousand enterprises spend over a million dollars on cloud/code rollouts, but existing problem classes are getting worse.

### AI Code Quality and the Poisoned Environment [05:43](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=343s)
AI-generated code quality is worse than human-generated code. Beyond that, the environment itself — skills, MCP servers, and shared tools — is being poisoned, both intentionally and unintentionally, with malware injected. Every company wants to be an AI company, and building with agents and models creates an entirely new threat surface on top of the existing one.

### Real Backlog Data and the Growing Threat [07:18](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=438s)
Nair shares data from 4,800+ customers: their vulnerability backlog grew 108% quarter over quarter in the past year. Despite the best agents and industry efforts, vulnerabilities are growing, not shrinking. Novel exploits like LLM-based attacks are really existing vulnerabilities chained together in the new surface, creating a much bigger blast radius. He references a Five Eyes statement from the prior week warning that AI will bypass cybersecurity systems in months, not years.

### Untrusted Agentic Environments and Agent Behavior [08:52](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=532s)
Nair presents benchmark data with QR codes linking to underlying studies. Key findings: vulnerable code from latest models is significant; over a third of skills across open-claw, Claude, Codex, and others contain malware or vulnerabilities — "three lines of English" can bring a system down. MCP servers have very little security built in; Snyk highlighted a GitHub MCP server exploit a year ago, and some customers immediately shut down all MCP servers in response. On behavior, agents in Fortune 100 environments have autonomously created copies of PII data in untrusted databases, creating unknown attack surfaces outside any enterprise security coverage.

### You Can't Govern What You Don't Know Exists [10:56](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=656s)
Real data from 3,000+ customers shows that for every model found in a repository, there are three times more agentic components. The full landscape must be mapped because risk spans multiple layers. Once found, independent data verification is needed to assess how risky each component is.

### Red-Teaming Results: Model Performance Variance [11:27](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=687s)
Snyk's risk benchmark tests new models and components as they emerge. The first two frontier models (unnamed) did well on PII extraction — zero extraction with Snyk's attacks. A third "hot new model" (rhyming with "LLM") allowed 100% PII extraction. However, on a decision-override test, the frontier models performed worse while the open model resisted 0% of the time. This variance means you must know what to use, when, and how — and these dynamics change constantly.

### Generator-Validator Separation: Fresh Benchmark Data [12:00](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=720s)
Nair shares research published the day before the talk. When asking the latest models (not yet publicly available) to find the same vulnerability five times, only 50% were found across the five tests. Versus a deterministic check, only 75% of issues were found, with a 40% F1 score. The point: probabilistic systems are useful but cannot be solely relied upon for enterprise security. They should be used for what they're good at, alongside deterministic checks.

### Snyk's Current Solutions [13:33](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=813s)
Nair outlines what Snyk has built so far: preventing new issues from entering the agentic loop (Snyk Studio), and package health checks that prevent agents from selecting vulnerable or malware-infected packages. For the existing vulnerability mountain, Snyk's remediation capabilities helped Labelbox reach zero backlog, and a Max 7 company remediated 16,000 critical issues in one week using a remediation agent. Snyk GA'd its agentic dev security offering the day of the talk, covering environment, output, skills, MCP servers, and behavior of coding agents like Cursor Cloud and Codex.

### Live Demo: Package Health and Skill Risk Assessment [15:37](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=937s)
Ezra takes the stage for a rapid-fire demo. The first demo asks Claude to generate a CLI tool for QR code creation, using Snyk's package health check tool. Two packages are compared: `qr_code` (healthy, actively maintained) and `qr_image` (no current CVEs but unmaintained, last released 10 years ago). The second demo assesses a competitive analysis skill, finding four risk findings: it echoes authorization headers (dangerous), pulls live content from Reddit and Twitter, and critically, fetches a YAML file from the internet for execution logic — meaning a third-party website change could introduce an exploit even if the local skill file is unchanged. Conference Wi-Fi issues forced a switch to a recorded demo.

### Evo and the AI Security Engineer Vision [20:49](https://www.youtube.com/watch?v=1EZdpEhwmNc&t=1249s)
Nair wraps up by describing Evo, Snyk's system introduced late last year, built around the OODA loop (Observe, Orient, Decide, Act) — the same training concept used for fighter pilots. The vision is to empower AI security engineers with powerful systems and tools, making them "10x superpowered" just as AI engineers became 10x engineers. He acknowledges unsolved problems around coordinating multiple agents, shared memory, and harnesses, and frames Evo as an open vision built with the community. He invites attendees to a community jam at Snyk's SF office.
