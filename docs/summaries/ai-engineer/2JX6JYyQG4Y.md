## TL;DR

Antje Barth from Amazon AGI Lab introduces "perception agents" — AI agents that perceive rendered screens the way humans do, enabling them to verify their own work, accept precise visual annotations as input, and collaborate in real time. She demos two open-source tools (annotation and verification) and shows how wearable devices can feed meeting transcripts directly into agent workflows, arguing that shared perception — not bigger brains — is the key to agent reliability and trust.

## Key Takeaways

- Agents today can perform individual steps (clicking, typing, API calls) reliably but still fail at end-to-end workflows because real work lives "in the seams" between different applications and systems.
- Reliability, not capability, is the current bottleneck — agents need to succeed in "the nines," not 60–80% of the time, before users can trust them.
- Coding agents achieved reliability first because code is verifiable (you can run and test it), but most knowledge work has no equivalent unit test.
- Humans handle messy, unverifiable work through shared context — looking at the same screen and collaborating in real time.
- Perception agents complete the perceive → plan → act loop on computer use by reading rendered screens, confirming their own output, and reacting while you're still working.
- Perception agents don't need APIs or backend access — they work off the same rendered pixels and structure that humans see.
- Visual annotation (pointing at elements) is a more precise, less lossy input signal than long text descriptions.
- Amazon AGI Lab open-sourced two pieces of a perception agent harness: an annotation Chrome extension and a verification tool that checks work against design specs.
- Perception extends beyond screens — wearable devices (like Friend's "B" devices) can capture meeting transcripts and feed them directly into agent workflows.
- The team is building the rest of the harness in the open and actively soliciting community feedback and contributions.

## Detailed Breakdown

### The Gap Between Capability and End-to-End Work [00:36](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=36s)

Barth opens by reflecting on how far agents have come in just a year — from struggling to find and click a button to driving browsers and desktop apps. But she argues that clicking was "the easy part." The real unsolved problem is end-to-end work. She gives the example of onboarding a new team member: setting up accounts, adding them to Slack, booking intros, ordering laptops — a process touching five different systems with no single owner. Agents can perform each individual step but still fail to complete the full workflow because the real work lives "within the seams" of all those applications.

### Capabilities vs. Reliability [02:08](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=128s)

We taught computers to use computers — clicking, scrolling, typing, calling APIs, filling forms — and got those individual steps reliable enough to string into workflows. But that's not enough to hand agents more of your work and walk away. The next hard problem is reliability, and without it there can be no trust. Barth asks the audience to gut-check: how often does an agent succeed end-to-end? Maybe 60–80%? That sounds fine until you realize a one-in-four failure rate means the agent might delete a database. For real trust, you need reliability "in the nines."

### Why Coding Was Solved First [04:46](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=286s)

Barth traces the rapid evolution of coding tools — from autocomplete, to writing functions, to coding agents that write code and open pull requests autonomously. She notes that we used to read every generated line; now code flies by at a pace no one can keep up with, and we've largely stopped checking. The reason coding achieved reliability and trust first is that code is verifiable: you can run it, test it, and confirm it worked. But most knowledge work is messy — did the report land? Is the design on brand? Did the agent understand what I meant? There's no unit test for those questions, so verification "hits the wall" where most work actually lives.

### How Humans Handle Messy Work [07:24](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=444s)

Humans successfully navigate messy, cross-system work every day by collaborating — jumping on a Zoom call, looking at the same screen, pointing at things, and solving problems in minutes. None of that work is directly verifiable, yet we do it constantly. The key insight is shared context: when you and a colleague look at the same screen, you have far less explaining to do. Barth argues agents don't necessarily need a bigger brain — they need this shared context.

### The Perceive → Plan → Act Loop [08:59](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=539s)

Today's agents can see a screen and take actions, but when they fire off an action they typically move on without watching what happens or recovering from failures. Barth draws an analogy to robotics, where a robot perceives its environment, plans, and then acts — a loop that's missing in screen-based agents. The agent needs to perceive the rendered screen (not scrape the code behind it), understand layout and state, keep up in real time, and react while you're still working rather than forcing the turn-based prompt-wait-prompt rhythm we've grown used to with chatbots.

### Introducing Perception Agents [11:36](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=696s)

Barth names this category "perception agents" and lays out three reasons they matter. First, they complete the loop on computer use: they can read the rendered screen and confirm their own output instead of just firing actions and hoping. Second, they don't need APIs or backend processes — they work off the rendered interface, which is critical because most everyday software exposes no APIs at all. Third, input flows the other way: instead of writing long text descriptions of desired changes, you can simply point at elements on screen and say "change this heading" or "double this font size" — a more precise, less lossy signal than text.

### Open-Sourcing the Annotation Tool [13:12](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=792s)

Barth announces that Amazon AGI Lab has open-sourced the first two pieces of their perception agent harness. The first is an annotation tool — a Chrome extension that lets you select elements on a screen by drawing around them or hovering and clicking. You can specify changes (e.g., "turn the heading red," "double the font size") and the tool captures the exact screen location, style elements, and your feedback into a complete summary that the agent can act on, eliminating back-and-forth.

### The Verification Tool [15:17](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=917s)

The second open-sourced piece is a verification tool. You describe design rules in a `design.md` file — colors, components, layout — and the agent converts them into rules if not already written. It performs two kinds of checks: a visual check (is everything on brand, correct layout?) and user-flow checks (it walks through the app like a real user, adding and deleting tasks). It then writes a report flagging what passed and what didn't, so you don't have to click through everything yourself at midnight.

### Beyond Screens: Wearable Perception [16:53](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=1013s)

Barth shares a live experiment from the conference: she and colleague Giovanni held a design meeting in a booth, both wearing "B" devices (sponsored by Friend) that transcribed their conversation. The transcript was pulled directly into the workflow as a detailed summary of discussed changes. With one click, those changes — yellow background, red heading, a new emoji — were applied to the website and verification kicked off automatically. The verification report confirmed the color scheme was within approved rules. Barth emphasizes this shows perception is "so much more than just the visual part."

### Building in the Open [19:30](https://www.youtube.com/watch?v=2JX6JYyQG4Y&t=1170s)

Barth stresses these two tools are just the beginning and the team is building the rest openly. She asks the audience to try the tools, break them, and provide feedback on what's missing and where they should go next — because "none of us get smart alone." She recommends a podcast by colleague Danielle Persik (a cognitive scientist leading the AGI/ACI team) on human-agent interaction patterns, and points to an upcoming talk by colleague Gaf Mishra at 1:30 in the computer use track ("From RL to IRL"). She invites attendees to visit the Amazon booth in the expo hall or check out their GitHub repos and website.
