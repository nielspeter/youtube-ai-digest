## TL;DR
Nick Ung and Asha from Lyft share their hard-won lessons from building and scaling a multi-agent customer support AI system, emphasizing that evaluations must be actionable, tied to business outcomes, and gated to launch decisions. They detail an end-to-end evaluation pipeline—from offline simulation with fine-tuned user personas to rigorous LLM-as-a-judge validation and continuous error analysis—providing a practical blueprint for teams moving agentic applications into production.

## Key Takeaways
- **Evals must gate something.** An LLM-as-a-judge score that doesn't block a launch or change a decision is not worth building.
- **Offline evaluation is essential before production.** Treat AI agent launches like traditional ML model launches: simulate multi-turn conversations, grade them, and enforce a launch gate so live users are never your test data.
- **Synthetic data should mirror reality.** Don't just prompt an LLM for 50 test queries; sample real production data and mutate criteria to cover golden paths and edge cases.
- **Frontier LLMs make poor user simulators.** They're trained to be polite and verbose, whereas real support users are impatient and terse. Fine-tuning a user simulator on real verbatims makes evals harder but far more realistic.
- **Define user personas to ground simulations.** Personas like "bypasser," "refund seeker," or "AI skeptic" help the simulated user behave like actual customers.
- **Generic metrics are not actionable.** Pre-built metrics like "response helpfulness = 0.5" don't tell you what to fix. Collaborate with domain experts to define binary, task-success/failure rubrics tied to business goals.
- **Treat LLM judges as ML classifiers.** Label ~100 examples, split into train/dev/test sets, iterate on the judge's prompt using few-shot examples, and measure precision and recall against human ground truth.
- **Criteria drift is real.** Evaluation criteria should co-evolve with the model as you observe more data and grade more outputs—don't lock criteria upfront.
- **Add statistical rigor when it matters.** Use confidence intervals and proper sampling for shipping decisions or leadership reports; small sample sizes can make negligible gains look significant.
- **Close the loop with continual learning.** Feed error-analysis insights back into model learning (weights), context learning (knowledge, memory), and harness learning (prompts, tool schemas, routing).

### Introduction and Agenda [00:05](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=5s)
Nick Ung, a data science manager, and Asha, a team member, introduce themselves as Lyft employees who have spent one to two years building a customer-support AI agent. They frame the talk around building evaluations that "actually matter" and scaling a multi-agent system, previewing an agenda that covers offline evals, online evals, eval harness, and future plans.

### End-to-End Evaluation Pipeline [02:11](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=131s)
Nick walks through a high-level system diagram spanning development and production. In development, teams manage context, build retrieval pipelines, define tools, construct the agentic graph, and write system prompts. Before launch, agents must pass a rigorous offline evaluation—mirroring traditional ML practice—so live users aren't used as test data. In production, online evaluation kicks in with tracing tools, online graders, and a human-in-the-loop pipeline for error analysis and feedback.

### Three Common Reasons Evals Fail [05:51](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=351s)
Nick identifies why evaluation efforts typically fail: (1) grader scores aren't meaningfully gating anything; (2) LLM judges are too noisy, generic, and produce insights people don't trust or can't act on; and (3) when production regresses, there's no clear mechanism to catch it or assign owners to act on the insights.

### Offline Evaluation System and Taobench Inspiration [07:23](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=443s)
Lyft's offline simulator is inspired by **Taobench**, a paper from Sierra AI designed for customer-support agents. The setup pairs a LangGraph agent with a user LLM that generates multi-turn trajectories. The simulation is grounded by agent domain policy, user intent, user data/world state, and user personas—enabling realistic, representative test conversations.

### Deterministic Evaluators and Code Assertions [10:32](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=632s)
Beyond LLM-as-a-judge, Lyft uses deterministic evaluators resembling traditional unit tests. These code assertions check concrete outcomes—for example, whether the agent actually granted a concession when it was supposed to, or whether tool calls matched expected behavior—measuring instruction-following accuracy.

### Synthetic Data Challenges [11:37](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=697s)
Nick warns against the common shortcut of simply prompting an LLM to generate test queries. Instead, Lyft samples real production data and mutates criteria to cover golden paths and edge cases, ensuring the offline dataset closely resembles production traffic.

### Fine-Tuning a Realistic User Simulator [13:11](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=791s)
A major gotcha: frontier lab models are trained to be helpful, polite assistants, not frustrated, impatient Lyft users. The first pass of offline evals produced overly verbose, patient user simulations and a suspiciously high 90% pass rate. Lyft fine-tuned a user LLM on real user verbatims, which lowered scores but made evaluations meaningful and revealed room to handle "difficult" users. They also defined personas (bypasser, refund seeker, AI skeptic) to further ground behavior, citing a similar approach from a Microsoft paper.

### LLM-as-a-Judge: The Problem with Generic Metrics [17:53](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1073s)
Asha takes over to discuss the second common problem. Most teams use LLM judges to measure generic metrics—tool usage, helpfulness, naturalness, toxicity, etc. Lyft started with pre-built metrics from **DeepEval** but found them not actionable: a "response helpfulness of 0.5" gives no guidance on what to fix. These can serve as a baseline but shouldn't be core eval metrics.

### Building Actionable, Binary Metrics [19:26](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1166s)
Asha advocates collaborating with domain experts to frame evals around task success or failure with binary outcomes, which are easier to calibrate. She shares an example "education rubric" metric: the agent fails if it tries too many times to educate when it should have escalated, or escalates too soon without attempting education. Expected behavior passes; deviations fail.

### Validating the Judge Like an ML Classifier [21:00](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1260s)
To validate an LLM judge, treat it as a binary classifier. Label ~100 examples with pass/fail, split into train/dev/test sets (with different percentages than traditional ML since no weights are trained). Use train examples as few-shot prompts, iterate the prompt against the dev set, and validate against the test set to check for overfitting. Measure precision and recall against human ground truth.

### Criteria Drift and Co-Developing Evals [23:39](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1419s)
Evaluation criteria shouldn't be fixed upfront. As you grade more outputs and observe model behavior, your sense of quality evolves. Criteria should co-develop with the model—refining metrics and re-evaluating the judge as new examples surface. Evals decoupled from model observations risk measuring the wrong things.

### Statistical Rigor and Confidence Intervals [24:47](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1487s)
Asha urges adding statistical rigor to reported numbers. Bare point estimates can be misleading: a 4-point gain (84% vs. 88%) on just 50 samples may not be real. Confidence intervals, calibration, and proper sampling make numbers meaningful—especially for shipping decisions or leadership reports. Reserve expensive rigor for moments that gate something, but every score needs an interval.

### Eval Anti-Patterns and the Importance of Looking at Data [26:57](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1617s)
Asha shares a non-exhaustive list of eval anti-patterns and highlights the most critical: ignoring the data. Without examining raw traces, you can't create meaningful criteria or labels; without labels, you can't evaluate your judges; without evaluating judges, you don't know if your pipeline works. Data inspection is the foundation.

### Continuous Error Analysis Loop [27:59](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1679s)
Lyft runs a continuous error-analysis loop: deep-dive into raw traces, pinpoint failure modes, keep only metrics that change a decision (removing noise), form a fresh premise, and repeat on a regular cadence (weekly or bi-weekly). This is not a one-off audit.

### Tracing and Annotation Queues [29:01](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1741s)
Tracing underpins everything. Tools like **LangSmith** and **Langfuse** log full graph executions—nodes, LLM inputs, tool calls, token usage, latency. Annotation queues provide a UI for domain experts to label traces and give feedback without parsing raw JSON. These labeled traces feed offline datasets and validate judges with ground truth.

### Closing the Loop: Continual Learning [31:07](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1867s)
Nick returns to explain how eval insights feed back into three learning modes: **model learning** (post-training, updating weights, custom LLMs), **context learning** (improving documents, stored memories, tool outputs the agent sees), and **harness learning** (updating system prompts, tool schemas, control flow, routing, retries). Error analysis directly informs all three.

### What's Next: Eval Harness and Post-Training [32:42](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=1962s)
Lyft's current offline simulator is scattered across notebooks and repos. They're investing in a systematic, config-driven **eval harness** stored as YAML files editable by engineers, analysts, and data scientists. The harness defines primitives (task, datasets, personas, LLM adapter, evaluator) and supports parallelism for tens of thousands of examples. Future work also includes post-training: fine-tuning models for specific support tasks and framing reward modeling for reinforcement learning.

### Eval Harness in Practice [34:17](https://www.youtube.com/watch?v=3z2uT5aDx_Y&t=2057s)
The eval harness lets teams define a config once and run evals indefinitely across the development lifecycle—locally during prompt tuning, at pre-commit hooks to catch regressions, and in CI/CD as regression and acceptance test suites. Nick closes by inviting the community to share learnings and brainstorm eval improvements.
