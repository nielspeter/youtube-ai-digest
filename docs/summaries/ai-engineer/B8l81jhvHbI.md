## TL;DR
Enterprise data agents fail not because of inadequate models or insufficient knowledge bases, but because of three structural problems: ambiguity, staleness, and preference. Ishita Daga proposes a hierarchical source-of-truth framework, a living context lifecycle with feedback loops, and acknowledges that capturing individual or team-level metric preferences remains an unsolved, open research challenge.

## Key Takeaways
- When agents give bad answers, the reflex to use bigger models or add more knowledge bases misses the root cause—structural problems in how context is organized and maintained.
- **Ambiguity**: Agents don't know which knowledge base, table, column, or definition to trust when multiple sources exist with unclear hierarchy.
- **Staleness**: Context degrades quickly as definitions, KPIs, and processes change, making static `.md` files and manually updated skills insufficient.
- **Preference**: Different teams legitimately calculate the same metric differently, and agents currently lack a robust way to route to the correct definition based on who is asking.
- For ambiguity, sources of truth should be structured as a hierarchy: semantic layer (cleanest, least flexible) → canonical tables (moderate flexibility) → database graph (messiest, most flexible).
- The semantic layer and canonical tables solve roughly 80% of ambiguity problems and should be implemented first; the database graph handles the remaining 20% but is costly to maintain.
- For staleness, Daga recommends a "context lifecycle" with two components: embedding living data sources (GitHub, CRM, Tableau, DBT) and building a feedback loop that logs correction events.
- Evaluation is critically underused—teams should curate human-annotated eval suites or automate comparisons against recent question-answer pairs to track agent performance over time.
- For preference, current approaches (semantic layers, agent memory like mem0 or `memory.md`) store definitions or preferences but cannot intelligently route to the right metric based on user identity.
- The ultimate goal for preference is a "hive mind" for the data agent that embeds individual and team preferences and routes accordingly—an open problem requiring further research.

## Detailed Breakdown

### Introduction and the Misdiagnosis of Agent Failures [00:01](https://www.youtube.com/watch?v=B8l81jhvHbI&t=1s)
Ishita Daga introduces herself as a machine learning engineer at Tesla building enterprise agents. She frames the talk around why agents fail, noting that the common reflex—upgrading to a bigger or newer model with more context, or adding more knowledge bases in the form of `.md` files, MCP servers, or plugins—is a misdiagnosis. While these are fair solutions, they don't address the structural problems inherent in the data agent itself.

### Problem One: Ambiguity [01:04](https://www.youtube.com/watch?v=B8l81jhvHbI&t=64s)
The first structural problem is ambiguity. Agents face a landscape of multiple knowledge bases, tables, columns, and data sources without clear guidance on which holds the source of truth, which is cleanest, or which is most appropriate for a given question. Daga argues that agents cannot weight all knowledge bases equally—there must be structure governing which source to consult.

### A Hierarchical Source of Truth [03:08](https://www.youtube.com/watch?v=B8l81jhvHbI&t=188s)
Daga proposes that the source of truth is a hierarchy, ranging from the cleanest and least flexible to the messiest but most flexible. She divides sources into three buckets. The **semantic layer** is the best source: a curated, single layer containing all KPI definitions, metric calculations, and business definitions. The agent simply finds the closest KPI and references its data points. **Canonical tables** offer moderate flexibility—the agent receives a list of parametric queries and can choose or adapt filters to answer a wider set of questions. The **database graph** is the most flexible but hardest to maintain: it connects every table to its columns and metrics, creating an enormous graph the agent can query. Daga recommends starting with the first two layers, which solve about 80% of problems, and reserving the database graph for the remaining 20%.

### Problem Two: Staleness and the Context Lifecycle [05:43](https://www.youtube.com/watch?v=B8l81jhvHbI&t=343s)
Staleness is the second structural problem. Context rots quickly—definitions, KPIs, and processes change constantly, and keeping `.md` files or skills manually updated is unsustainable. Daga's solution is a "context lifecycle" with two components. The first is embedding **living data sources**—always-updated, well-curated sources such as GitHub, CRM tools, Tableau, or DBT semantic layers—that serve as mandatory, frequently refreshed inputs. The second is a **feedback loop**, which many teams overlook. Every correction event (e.g., "this definition is wrong," "there's a new filter") must be logged and used to update the agent's context. After capturing events, teams must evaluate the agent, either through human-annotated eval suites or automated comparisons of recent question-answer pairs. Daga stresses that without evaluation, teams have no visibility into agent performance degradation.

### Problem Three: Preference [08:51](https://www.youtube.com/watch?v=B8l81jhvHbI&t=531s)
Preference is the most subjective and least solved problem. Different teams may legitimately calculate the same metric differently. Daga gives a concrete example: Team A calculates average milestone time from the completion of the previous milestone to the completion of the current one, while Team B measures from the start of the current milestone to the start of the next. Both are valid, yet they yield very different numbers. The agent needs to understand who is asking and route to the correct definition accordingly.

### Current Approaches to Preference and Their Limitations [10:24](https://www.youtube.com/watch?v=B8l81jhvHbI&t=624s)
Daga outlines two existing approaches, both insufficient. Storing multiple metric definitions in a **semantic layer** lets users prompt the model to choose, but it doesn't persist user preference and reintroduces ambiguity—the agent still doesn't know which definition is right without explicit prompting. **Agent memory** tools like mem0 or `memory.md` files can store preferences but cannot distinguish between competing metric definitions or intelligently route based on user identity. What's actually needed is a mechanism to route the agent to the right metric based on which team or individual is using it—something Daga describes as creating a "hive mind" for the data agent. She acknowledges this remains an open research problem.

### Conclusion [11:29](https://www.youtube.com/watch?v=B8l81jhvHbI&t=689s)
Daga recaps the three problems—ambiguity, staleness, and preference—and emphasizes that solving them requires better structure and context management, not just bigger models. Ambiguity and staleness have actionable frameworks (hierarchical sources of truth and context lifecycles), while preference remains an open-ended challenge requiring deeper research into embedding individual and team-level preferences into agent behavior.
