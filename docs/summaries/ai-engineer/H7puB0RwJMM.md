## TL;DR
Daniel Chalef presents how Zep AI and their open-source framework Graffiti solve the provenance problem in LLM-built knowledge graphs. Because LLMs non-deterministically synthesize data from multiple sources, tracing facts back to their origins requires modeling lineage directly into a graph data structure—enabling compliance, veracity checks, debugging, and safe deletion of derived data.

## Key Takeaways
- LLMs synthesize data non-deterministically, destroying the "paper trail" of how extracted facts and structured records were built.
- Provenance—tracing how an artifact was built and why—is essential for legal compliance, debugging, trust evaluation, and determining what to delete.
- Simple source IDs fail because LLM pipelines merge entities, mutate facts, and constantly change the underlying store.
- Graffiti models provenance as a knowledge graph: source data ("episodes"), entities, and edges (facts) are all nodes, making lineage a graph walk.
- When entities merge or facts are contradicted by new data, lineage links must survive mutation to avoid silently dropping source information.
- Metadata projection allows tags (e.g., "EHR-verified") applied at ingestion to propagate to all derived entities and facts, enabling veracity filtering.
- Different business rules apply to similar-shaped facts: a drug allergy fact may require only one verified source, while a consent fact may require all sources verified.
- For privacy compliance (e.g., right-to-be-forgotten), a fact is only deleted if no remaining supporting episodes survive after source deletion.
- Lineage must be engineered into the data structure (the graph), not logged after the fact.
- Graffiti uses a mix of LLMs and traditional NLP/IR techniques (sim hash, entropy analysis) to reduce the cost and latency of graph construction.

## Detailed Breakdown

### The Provenance Problem with LLMs [00:01](https://www.youtube.com/watch?v=H7puB0RwJMM&t=1s)
Chalef opens by explaining that LLMs excel at pulling data from many sources but do so non-deterministically. They interpret and synthesize data into summaries, extracted facts, or structured records that may not appear verbatim in any source input. This synthesis destroys the paper trail of how outputs originated. Provenance—tracing how an artifact was built and why—is needed for legal compliance, debugging, deciding which sources to trust, and determining which artifacts to delete. Solving this at scale is a significant engineering challenge.

### Graffiti, Zep, and the Healthcare Failure Mode [01:04](https://www.youtube.com/watch?v=H7puB0RwJMM&t=64s)
Chalef introduces Graffiti, an open-source temporal graph framework, and Zep, enterprise agent memory infrastructure built on Graffiti. Customers derive agent memory from chat, voice transcripts, email, and business data, and they struggle with questions like "Where did this fact come from?" and "What is its veracity?" He presents a healthcare failure mode: an agent retrieves a confident fact—"Patient has a penicillin allergy"—synthesized from three sources (an EHR record, a PDF lab report, and a patient-typed AI intake chat). If the agent presents this to a doctor without indicating the source was the patient themselves, it could mislead the doctor.

### Why Source IDs Don't Work [03:07](https://www.youtube.com/watch?v=H7puB0RwJMM&t=187s)
A simple approach—storing a source ID on each fact—works in structured data warehouses where pipelines copy or mutate data deterministically. But LLM context pipelines break this model. When you prompt an LLM with several sources, many facts are synthesized from multiple sources. Entities like "J. Smith" and "John Smith" merge into one identity. New data invalidates old facts. The store keeps changing underneath your pointer, making append-only logs hard to manage at scale. Lineage must be an evolving set that survives mutation.

### Modeling Provenance as a Knowledge Graph [04:08](https://www.youtube.com/watch?v=H7puB0RwJMM&t=248s)
Sets of links between facts and sources can be modeled as graph relationships. In Graffiti, source data becomes "episodes" (nodes), entities are extracted from episodes, and edges between entities represent facts. A graph triple—two entities and an edge—can be hydrated as a fact like "Patient has a penicillin allergy." Tracing a fact to its source becomes a simple graph walk. However, keeping lineage correct as the graph changes is hard: when entities merge, the merged entity must retain all source links from both originals, or lineage is silently lost.

### Handling Contradictions and Invalidated Facts [05:09](https://www.youtube.com/watch?v=H7puB0RwJMM&t=309s)
When new data contradicts existing data, Graffiti captures this lineage too. An invalid date is added to the mutated edge, and the source episodes that caused the mutation are noted against the fact. This ensures the system records not just what changed but which new sources drove the change.

### Metadata Projection for Veracity Filtering [06:13](https://www.youtube.com/watch?v=H7puB0RwJMM&t=373s)
Graffiti supports metadata projection to model classifications spanning many episodes. In the healthcare example, episodes from an EHR record are tagged "EHR" at ingestion. All subsequent entities and facts derived from those episodes inherit the tag. If an agent needs only facts from verified clinical sources, it filters by the appropriate tag during the graph walk—one tagging action at ingestion supports ongoing veracity evaluation.

### Business Rules for Multi-Source Facts [07:18](https://www.youtube.com/watch?v=H7puB0RwJMM&t=438s)
When a fact has three or more parent episodes, veracity policies can differ. For a penicillin allergy flag—a potential life-and-death situation—even one unverified source should block a prescription. For a consent-on-file fact, every parent episode must be verified; operating on unverified consent is the risk. The facts have similar shapes but opposite policies. Graffiti exposes which episodes carry which tags, but the agent must apply the business rules—this logic is situational and not baked into the graph.

### Privacy Compliance and Deletion [08:52](https://www.youtube.com/watch?v=H7puB0RwJMM&t=532s)
Lineage is critical when source data must be deleted due to retention policies or right-to-be-forgotten requests. If only some source data needs deletion (e.g., the intake chat data), Graffiti's model handles it gracefully: the allergy fact survives because two parent episodes still support it, but the contact-preference fact is deleted because it was derived solely from the deleted source. The rule is simple—a fact is only deleted if no remaining episodes support it—and it is enforceable because the links exist.

### Summary and Cost Considerations [10:29](https://www.youtube.com/watch?v=H7puB0RwJMM&t=629s)
Chalef summarizes: deriving context is lossy and generative, so lineage must be engineered into the data structure (a graph), not logged afterward. Graffiti keeps sources verbatim and links everything derived from them back to the source. Benefits include compliance, veracity verification, debugging, and deletion determination. He notes that graph construction is expensive the way Graffiti does it, and the team has invested significantly in reducing cost and latency, a topic he offers to discuss in Q&A.

### Q&A: Edge Weight Mutations [12:41](https://www.youtube.com/watch?v=H7puB0RwJMM&t=761s)
An audience member asks about mutating the graph to account for weight changes in relevancy across many edges. Chalef explains that this structure is built into Zep rather than Graffiti, using a separate data structure from the graph for tracing provenance across edges in that particular scenario.

### Q&A: Graph Construction and the Add Episode Pipeline [13:46](https://www.youtube.com/watch?v=H7puB0RwJMM&t=826s)
Asked whether the agent creates edges, edge types, and entities itself, Chalef explains that Graffiti supports vector similarity search, full-text search, and graph relational operations (like BFS) depending on the underlying graph database. Episodes are nodes on the graph, as are derived artifacts, because they too need lineage. The `add episode` method runs a complex pipeline: structured extraction of entities and relationships (candidate fact triples structured as subject-verb-object), followed by deduplication and deconfliction. For example, "Daniel loves Adidas shoes" is invalidated when "Daniel returned shoes" and "Daniel was unhappy about Adidas" are learned. Graffiti uses LLMs where necessary but deploys traditional NLP and IR techniques (sim hash, entropy analysis) for deduplication—cheaper, faster, and more deterministic.

### Q&A: File-Based Memory and Provenance [17:29](https://www.youtube.com/watch?v=H7puB0RwJMM&t=1049s)
An audience member asks about file-based memory paradigms (markdown, wikis, knowledge bases). Chalef argues that markdown suffers from provenance problems: when lines in a file are mutated, understanding the lineage of those changes is very difficult. In multi-agent, multi-user, and multi-source scenarios, markdown files are challenging to manage at scale. They work well for desktop or single-user/single-agent scenarios but break down for enterprise problems, particularly provenance.

### Q&A: Fact Extraction Method [19:04](https://www.youtube.com/watch?v=H7puB0RwJMM&t=1144s)
Asked how facts are created—whether the LLM summarizes conversations at every turn—Chalef says Graffiti uses single-shot extraction to extract entities, relationships, and facts cheaply. A reflection step ensures retrieved facts are accurate and adds richness to lineage by capturing not just that something changed but why it changed. This is split partly between Graffiti and Zep.
