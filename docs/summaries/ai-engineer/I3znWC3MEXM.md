# It's 10pm. Do You Know Where Your Agents Are? — Kim Maida, Keycard

## TL;DR
Kim Maida illustrates the dangers of giving AI agents broad, persistent API keys and demonstrates how OAuth 2.0 Token Exchange (RFC 8693) can be used to implement secure, task-scoped access control. By introducing a Security Token Service into the agent execution path, organizations can ensure agents act only with short-lived, least-privilege permissions while maintaining a clear chain of identity and policy-backed human-in-the-loop oversight.

## Key Takeaways
- Agents equipped with broad API keys are effectively overprivileged, able to take destructive or costly actions without proper authorization.
- Human-in-the-loop alone is insufficient for security; humans suffer from consent fatigue and make mistakes, especially during off-hours.
- The agentic execution path—spanning the runtime, MCP client, MCP server, and resource API—offers specific points where access control can be enforced.
- RFC 8693 (OAuth 2.0 Token Exchange) is an existing, open standard that can secure agent access without requiring entirely new protocols.
- A Security Token Service (STS) acts as an authorization server that verifies identities, evaluates governance policies, and issues short-lived, ephemeral tokens.
- Token exchange introduces a chain of identity: the agent's identity, the delegating user's identity, and the user's level of access are all known and logged.
- Access tokens are scoped to a single tool call, audience-restricted to a specific MCP server, short-lived, and never stored—minimizing the risk of credential leakage or replay attacks.
- Policy is evaluated *before* a credential is minted; if a request violates policy, the token is never created, leaving nothing to steal or misuse.
- Human-in-the-loop approvals can be backed by real policy, preventing users from authorizing actions beyond their actual permissions.
- This approach works with off-the-shelf agents, custom builds, CLIs, third-party MCP servers, and emerging technologies.

## Detailed Breakdown

### The Problem with Overprivileged Agents [00:01](https://www.youtube.com/watch?v=I3znWC3MEXM&t=1s)
Maida opens with a nostalgic reference to public service announcements asking parents if they know where their children are, drawing a parallel to the current state of AI agents. She notes that agents are being entrusted with increasing responsibility, often given API keys and let loose to accomplish tasks. While this seems fine initially, it creates significant risk because agents become overprivileged and can act freely on decisions that may not align with user intent.

### Incident Management Agent Demo [01:36](https://www.youtube.com/watch?v=I3znWC3MEXM&t=96s)
Maida walks through a demo of an incident management agent working late at night. The agent triages several issues: a hardware failure (escalated), an expiring TLS certificate (renewed using a cloud hosting API key), a broken billing database (the agent drops the database based on documentation but cannot verify backups), frozen production processes (the agent takes prod offline using the same kitchen-sink API key), and site failures (the agent scales up, incurring spend). The demo highlights how a single overprivileged API key allows the agent to take increasingly risky and irreversible actions.

### Why Human-in-the-Loop Isn't Enough [04:52](https://www.youtube.com/watch?v=I3znWC3MEXM&t=292s)
Agents with API keys are overprivileged and can act even under supervision, as anyone who has panicked trying to stop an agent mid-task knows. The problem worsens with unsupervised agents because agents inherently want to be helpful and will use all available permissions. Maida argues that human-in-the-loop is insufficient because we spent decades solving access management for humans, and relying on a tired or consent-fatigued person to gate every action is not a reliable security boundary.

### The Agentic Execution Path [05:54](https://www.youtube.com/watch?v=I3znWC3MEXM&t=354s)
To identify where to introduce access control, Maida maps the agentic execution path: a user submits a prompt to a runtime (e.g., a CLI, SDK, or app), which calls the model. The model proposes tool calls dispatched by the MCP client to the MCP server, which executes tools and calls the resource API. Results flow back through the loop until the model returns a final answer. This path reveals several places where real access control can be implemented using open standards.

### Token Exchange and RFC 8693 [07:29](https://www.youtube.com/watch?v=I3znWC3MEXM&t=449s)
Maida introduces RFC 8693 (Token Exchange), an extension of OAuth 2.0, as the spec to address agent access. She recaps the core problems: credentials aren't attributed to identities, agents have unrestricted permissions, and human-in-the-loop is unreliable for autonomous agents. The solution involves an authorization server (Security Token Service) that verifies identities and issues tokens. The user first logs in and consents to delegate a subset of their permissions—the first narrowing of access. The authorization server issues a subject token identifying the user and their access level.

### Building the Identity Chain [09:00](https://www.youtube.com/watch?v=I3znWC3MEXM&t=540s)
To support token exchange, an OAuth client capable of executing code is needed—this could be a gateway between the MCP client and a third-party MCP server, a custom agent app, or a CLI wrapper. The agent loop runs, the model proposes a tool call, and the runtime authenticates with the STS using its own client credentials or workload identity. It also sends the subject token containing the user's identity and access level. This creates a token exchange request for permissions to access the MCP server for that specific tool call only. Three key pieces of information are now present: the agent's identity, the user's identity, and the user's level of access.

### Policy Evaluation and Ephemeral Tokens [10:33](https://www.youtube.com/watch?v=I3znWC3MEXM&t=633s)
Governance policy is evaluated against the requested access—who is asking, for what resource, on whose behalf. If the delegation chain and requested access are within policy, the STS issues an access token for the downstream resource. This token has an audience restricting it to the target MCP server, is short-lived (often expiring in minutes), and is ephemeral (never stored). The MCP client uses it as a bearer credential, the MCP server validates it, calls the resource, and discards the token immediately. The loop repeats until the model returns the final answer.

### Secured Demo with Token Exchange [11:40](https://www.youtube.com/watch?v=I3znWC3MEXM&t=700s)
Maida revisits the same agent and tickets, now with token exchange enabled. The operator authenticates via Google. The audit log shows significantly more detail: the agent's identity, who it is acting on behalf of, which MCP servers and downstream resources it contacts, and the specific scopes requested. For the certificate renewal, the agent requests only the scope needed to renew the certificate. For the billing database drop, policy evaluates the request before the credential is minted and blocks it—the credential never exists, so there is nothing to leak, replay, or steal. For restarting prod, the user is prompted for approval, but policy checks the user's role and denies the action because the user lacks the required role. For scaling, the user approves and policy permits it because the user has the necessary permissions.

### Benefits and Compatibility [15:52](https://www.youtube.com/watch?v=I3znWC3MEXM&t=952s)
The secured approach solves the earlier problems: user and agent identities are known, agent access is task-scoped and ephemeral, and human-in-the-loop is backed by real policy. Maida emphasizes that using open standards like token exchange ensures compatibility with off-the-shelf agents, custom agents, CLIs, third-party and proprietary MCP servers, MCP gateways, agent-to-agent communication, any OAuth identity provider, and emerging technologies like OpenClaw.

### Q&A: Placement of Security Barriers [17:31](https://www.youtube.com/watch?v=I3znWC3MEXM&t=1051s)
An audience member asks where the security barrier sits—between MCP and resource, or at both places. Maida clarifies that the authorization server sits between the runtime and the MCP server. The runtime authenticates to the STS and requests scopes for the specific tool call. If the request exceeds what is allowed for that tool call, the OAuth token is never sent downstream. The STS ensures the agent only receives tokens scoped to what is appropriate for each individual call.

### Q&A: Enterprise Adoption and Scope Definition [20:25](https://www.youtube.com/watch?v=I3znWC3MEXM&t=1225s)
Asked about enterprise resistance to adopting new protocols, Maida notes that RFC 8693 is not actually new—it has existed for some time and can be combined with newer specs. On the question of how granular to define OAuth scopes for MCP servers, she recommends starting with the scopes already defined on the resource server, as those represent the user's baseline access. Additional scopes for governing tool calls on custom MCP servers can be layered on top or passed through.
