## TL;DR
May Walter from Thundra presents an agentic workflow that continuously and autonomously optimizes application performance by analyzing production runtime data, identifying high-ROI opportunities, and generating verified fixes for human review. The system bridges the gap between production-level metrics and function-level code context, enabling coding agents to surface and fix performance issues that engineering teams would otherwise never proactively investigate.

## Key Takeaways
- **The investigation black box:** Performance issues are often ignored because the research phase to understand them is unpredictable, taking anywhere from an hour to a week, leading to a reactive "crisis mode" cycle.
- **Automating the investigation:** By running weekly agentic workflows with real production context, teams can automatically surface scored, high-ROI performance opportunities without stopping feature work.
- **Vendor-neutral infrastructure:** The system was deliberately built to be vendor-neutral regarding compute, harness, and models, allowing flexibility and easy maintenance over time.
- **Prod-to-code context:** A major breakthrough was mapping production metrics (service/endpoint level) to function-level context, allowing coding agents to reason over code using real runtime data.
- **Overcoming agent pitfalls:** Early challenges included "plausible unverified" fixes, lazy fixes (e.g., just catching exceptions), and complex database queries, which were solved through structured skills and methodology.
- **Human-friendly reporting:** Instead of overwhelming engineers with dozens of automated PRs, the system sends prioritized, readable reports to Slack, building trust and appetite for fixes over time.
- **Context over cleverness:** Providing the right context and skills to agents yields more reliable results than relying on the model's inherent intelligence alone.
- **Agentic engineering vs. coding with agents:** Autonomous workflows require 80-90% trust and accuracy, which is dramatically different from using an AI assistant interactively in an IDE where a human can steer 80% accurate outputs.

## Detailed Breakdown

### The Performance Investigation Black Box [00:01](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=1s)
May opens by describing a familiar scenario: a product manager reports a slow page, engineering estimates investigation could take "between an hour and a week," and only one person ("Dave") knows the legacy code. This highlights how the unpredictability of the research phase prevents teams from proactively optimizing performance, leading to a reactive cycle of ignoring issues until they become crises.

### AI Adoption and Software Delivery Instability [02:06](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=126s)
Referencing Google's 2024 DORA metrics, May notes that AI adoption has boosted individual effectiveness but also caused software delivery instability—things break more often, even though team throughput hasn't improved as much as expected. The premise is that if AI can help us build and fix faster, we can close that gap.

### The Vision: Automating the Performance Sprint [03:37](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=217s)
The core idea is to automate the investigation phase that teams typically skip. By running weekly with real production context, an agentic workflow can flag high-ROI opportunities—things that are both easy and impactful—essentially automating what a performance sprint does every few months.

### Infrastructure Requirements and Choices [05:10](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=310s)
The team needed vendor-neutral infrastructure for compute, harness, and models, with secure tool calls, trigger systems (scheduled runs or webhooks for SLO breaches), and easy maintainability. They chose GitHub Actions for orchestration, Claude Code as the agent, and used MCP to capture runtime intelligence, sending results to Slack.

### The Workflow: From Detection to Verified Fix [06:42](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=402s)
The workflow captures production traces, queries, and latencies, analyzes them with the agent, and scores opportunities based on business impact and frequency. Crucially, the agent doesn't just suggest fixes—it implements them, reruns tests, and verifies the runtime impact before handing a human-readable report to a person for review.

### Early Hurdles: Plausible Unverified, Complex Queries, and Lazy Fixes [08:15](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=495s)
Three major challenges emerged: fixes that sounded correct but didn't actually work in production ("plausible unverified"), complex query patterns (especially with ClickHouse), and "lazy fixes" where the agent would simply catch an exception rather than investigate root cause. These were addressed through methodical playbooks mirroring how a senior engineer would investigate.

### The Context Problem: Too Much and Too Little [10:19](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=619s)
Runtime context often presents two simultaneous problems: too much low-signal data that's hard for agents to reason over, and too little of the specific traces and logs needed to investigate. Additionally, service-level metrics (CPU, memory, P90s) don't map to function-level code, creating a language gap between what agents reason over and what production observability tools report.

### Prod-to-Code: Bridging the Gap [11:21](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=681s)
The solution, called "prod-to-code," maps production behavior to function-level context—the level at which coding agents reason. This connects endpoints, event consumers, and cron jobs to specific functions, outbound database calls, and LLM calls. Forensic evidence is captured only when requests exceed thresholds (like P99), keeping the signal-to-noise ratio high.

### Skills, Automations, and Performance Patterns [12:53](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=773s)
On top of the data layer, the team built a query language and a set of "skills" to create consistent agent behavior—such as investigating HTTP 500s or comparing memory spikes to baselines. Automations layer on top, targeting specific performance anti-patterns like artificial delays (timeouts, sleeps), N+1 queries, missing indexes, and sequential assignments, which are far easier to find with runtime context than static analysis alone.

### Human-Friendly Reporting Over PR Floods [15:30](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=930s)
Opening 80 automated PRs at once doesn't work—people are too busy. Instead, the system sends prioritized, human-readable reports to Slack, explaining the issue, the root cause, and the fix in a short gist. The team started with one issue at a time to build trust and habit, focusing on hot paths, business impact (e.g., payments, sign-ups), and risk, optimizing for highest ROI rather than absolute highest impact.

### Four Lessons Learned [18:38](https://www.youtube.com/watch?v=JJGbw4ggaFs&t=1118s)
1. **Define what matters:** Scoring and guardrails make the automation reliable; even if fixes are cheaper, they're not free, so worthiness must be established.
2. **Automating the unautomated:** This workflow doesn't just accelerate existing work—it automates a phase that simply didn't happen in engineers' day-to-day lives.
3. **Context over cleverness:** The right context and skills matter more than model intelligence, and models are already "good enough" for most cases.
4. **Agentic engineering vs. coding with agents:** Autonomous workflows require 80-90% trust, dramatically different from interactive IDE usage where a human can steer 80% accurate output in real time.
