# Skills are the New SDKs - Elvin Aghammadzada, DataRobot

## TL;DR
Elvin Aghammadzada argues that "skills" — progressively disclosed, metadata-indexed capability files — are becoming the new SDKs for AI agents, solving the context rot problem that plagues large context windows and bloated MCP tool definitions. Skills shift the enterprise moat from friction (hard to leave) to fluency (hard to leave because it's good), enabling a single general-purpose agent to adapt across industries by loading only the operational knowledge it needs at runtime.

## Key Takeaways
- **Context rot is real**: Research shows LLM performance degrades after ~25% of the context window is consumed; by 40% usage, agents enter a "dumb zone" regardless of model capability.
- **Infinite context is a "beautiful lie"**: Larger context windows don't equal better performance — every additional token is another place for the LLM to be misled or poisoned.
- **Skills use progressive disclosure**: Only metadata (front matter, <100 tokens) is loaded initially; the body (<5K tokens) and scripts load on activation, dramatically reducing context consumption compared to MCP tool definitions.
- **Friction moats vs. fluency moats**: Traditional SaaS moats rely on switching costs (friction); skills enable fluency moats where value compounds with each skill added, making users want to stay.
- **Teachability is the new enterprise checklist item**: Beyond security, compliance, and governance, platforms are now evaluated on how quickly an agent can acquire operational knowledge via skills.
- **Skills and MCP are complementary, not competing**: MCP handles remote execution, authentication, and resource isolation; skills handle reasoning, self-modification, and context-efficient capability loading.
- **Skills enable cross-industry agent reuse**: Instead of building domain-specific agents, one general-purpose agent can be adapted to manufacturing, supply chain, or other domains via skills.
- **Self-evolving skills are powerful but risky**: Agents writing their own skills can extend capabilities but introduce prompt injection risks, lack of isolation, and marketplace verification gaps.
- **The ecosystem is growing fast**: 26+ platforms support skills, with hundreds of thousands of skills already published and marketplaces emerging.
- **Skills are software**: They require versioning, evaluation, testing, and human authorship — LLM-generated skills have been shown to hurt performance.

## Detailed Breakdown

### Three Layers of AI Apps and Hidden Bugs [00:03](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=3s)
Every AI or agent application has three layers: the UI (what the user sees), the system prompt and tool descriptions (what the model sees), and the data schema / tool I/O (what data sees). Bugs typically live in the second and third layers, hidden from the user.

### The Infinite Context Window Myth [01:04](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=64s)
Frontier models promise 1M, 5M, or "infinite" context windows, but this incorrectly shapes thinking about RAG and MCP. People assume they can dump all documents or write 100 MCP tools and expect the LLM to pick the right one. In reality, longer context doesn't mean better performance — each addition is another opportunity for the LLM to be misled or context-poisoned.

### Context Rot and the Dumb Zone [02:07](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=127s)
A paper called "context rot" shows that after 25% usage of the context window (e.g., 256K of 1M tokens), performance starts to degrade. In long sessions with many tool calls, failures compound and a frontier model can perform like a poor model. Aghammadzada notes this is visible in Claude Code, which sometimes repeats mistakes from minutes earlier.

### Documentation Written for Humans, Not Models [03:10](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=190s)
Traffic to documentation websites from coding agents jumped from 10% to 50% in one year. Docs are written for humans — requiring intuition, follow-up questions, or Googling — but models expect a different format and lack those capabilities unless a context engine is built around them.

### Friction Moats vs. Fluency Moats [04:14](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=254s)
Historically, software moats came from controlling hardware, data, or integrations — making switching costs prohibitively high (friction moats). Today, tools like Claude Code can rewrite 100K–1M lines of code between languages in days, reducing switching costs. Skills create a new "fluency moat" where each skill compounds the user's experience, making the platform genuinely hard to leave because users like it. Fluency moats are offensive (make it great) rather than defensive (make it hard to leave).

### Teachability as the New Enterprise Checklist Item [07:20](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=440s)
Enterprise platform evaluation checklists include security, compliance, data governance, SLAs, logs, tracing, and integration. Aghammadzada argues "teachability" is the new item: how quickly can a new agent harness pick up operational knowledge encoded in a skill and reach the desired outcome?

### Context Engineering and the 40% Threshold [09:21](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=561s)
Context engineering is about minimizing information flow to the LLM, filling context with the right information at the right moment. A typical agent has system instructions, built-in tools, MCP connections, and user prompts. The "smart zone" is the first 40% of context; beyond that, responses degrade into the "dumb zone." If you load too many MCPs upfront, you may enter the dumb zone before even chatting.

### Progressive Disclosure: How Skills Solve Context Rot [11:23](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=683s)
A great context has instructions, external data, execution history, memory, and output schema — each adding complexity and creating an M×N problem. Instead of hardcoding everything, skills progressively load only what's needed at runtime. Only metadata is exposed initially, acting like a database index. This makes context consumption at least 10x smaller than MCP, where 15 MCP servers can consume 100K+ tokens just in tool definitions.

### Skills vs. MCP: Different Problems [16:07](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=967s)
Both skills and MCP come from Anthropic and aren't competing. MCP defines actions the agent should take without changing the agent's definition; skills help the agent think about problems and customize behavior, including self-modification. MCP runs in a separate environment (good for isolation, auth, heavy compute), while skills run on the agent's machine. A skill can even expose or wrap an MCP server, using progressive disclosure to manage access to resource-intensive MCP tools.

### Skills Structure: Two Readers, Three Levels [20:50](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=1250s)
Skills have two "readers": the agent's runtime (system prompt) reads the front matter (<100 tokens), and the LLM reads the body on activation (<5K tokens). Three levels exist: (1) front matter metadata, (2) markdown body loaded on demand, and (3) scripts — either executable (e.g., Python files Claude Code can run, returning only output to context) or non-executable example code.

### The Skills Ecosystem and Open Agent [22:25](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=1345s)
26+ platforms support skills (Claude Code, Codex, Copilots, Gemini CLI, etc.), with hundreds of thousands of skills and emerging marketplaces. DataRobot published an open-source agent skills repository. Open Agent — the most-stard repository, beating Winix and React — demonstrates impressive context engineering with sub-agents that keep context clean and self-evolving skills that can self-heal and write new skills. This shows you can build one good general-purpose agent and adapt it across domains via skills.

### Risks of Self-Evolving Skills [24:29](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=1469s)
Research shows LLM-generated skills can hurt performance — using more tokens and spending more time reasoning rather than helping. Risks include prompt injection (anyone can publish skills), lack of isolation (everything runs on the agent's machine, unlike MCP's process isolation), and unverified marketplaces reminiscent of early NPM.

### Closing Thoughts [26:01](https://www.youtube.com/watch?v=LC3-P7v3yoI&t=1561s)
Context is a budget — a limited resource requiring careful filtering. Longer context doesn't mean better; irrelevant files, web searches, and error messages drag agent attention and reduce reasoning power. Skills complement MCPs. A skill is only as good as the human who wrote it. Skills are software that can take weeks to build and should be versioned, evaluated, and tested.
