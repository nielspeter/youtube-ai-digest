## TL;DR
Moritz Johner from Form3 presents Patch Pilot, an automated CVE remediation system that gives AI agents production code access to patch vulnerable dependencies across thousands of repositories. The talk walks through the architecture, security boundaries, and sandboxing challenges of running autonomous coding agents with real production credentials—framing the core thesis that a useful coding agent is inherently a supply chain actor that needs the same guardrails as human engineers.

## Key Takeaways
- Traditional dependency patching tools (Dependabot, Renovate) only handle manifest-level bumps and can't reason about cascading changes needed to reach green CI.
- Patch Pilot uses a two-tier architecture: a deterministic Go controller for orchestration and credential-sensitive operations, and LLM agents for reasoning about CVE remediation and CI failures.
- The agent only modifies files on the filesystem—it never gets GitHub write access, CI trigger permissions, or other dangerous credentials. The deterministic layer handles commits, pushes, and PR creation.
- Prompt injection is an unsolved problem; the strategy is blast-radius limitation through architecture decisions rather than hoping for immunity.
- Giving an agent a Docker socket effectively breaks sandboxing, since a privileged container can escape and read host environment variables, memory, or plant SSH keys.
- Form3's current sandboxing approach uses Firecracker microVMs with separate kernels, combined with DNS/TCP forwarding through Vsocks to apply granular network policies.
- Agent retrospectives after each invocation help identify missing context, infrastructure issues, and repository complexity problems at scale.
- The agent sometimes outperforms humans—for example, removing pinned packages that were already fixed by a base image bump, which the speaker admits he would have missed.
- Existing agent sandboxes (Codex, Claude) are insufficient for Docker socket scenarios; open-source projects like microsandbox are promising but still early-stage.
- The blast radius of an agent is fundamentally an architecture decision: what's deterministic vs. agentic defines your security model.

## Detailed Breakdown

### The Problem with Traditional Dependency Patching [00:32](https://www.youtube.com/watch?v=LqLoYksJ6do&t=32s)
Johner opens by acknowledging that dependency patching is unglamorous but unavoidable at Form3's scale of thousands of repositories—a backlog that never empties. While Dependabot and Renovate exist, they were built for a simpler world where fixing a CVE means bumping a version in a manifest. Modern containerized environments are more complex: CVEs may live in OS packages in base images invisible to these tools, or in binaries pulled during Docker builds. Additionally, patches never happen in isolation—bumping a Go runtime may require bumping the linter, which introduces new linting rules that break the codebase. Traditional tools just bump the version, create a PR, and walk away, leaving engineers to clean up the mess. This makes patching not just a versioning problem but a reasoning problem.

### Introducing Patch Pilot and the Supply Chain Thesis [03:08](https://www.youtube.com/watch?v=LqLoYksJ6do&t=188s)
Patch Pilot was built to look at a CVE, understand its full surface, and figure out everything that needs to change to reach green CI. When Infosec asked whether this automation was a supply chain incident waiting to happen, Johner frames the thesis: a useful coding agent with production credentials is a supply chain actor, just like a human engineer, and should have similar guardrails. He emphasizes this is a real case study from production, not theory, and that Infosec's pushback was justified.

### Architecture: Two-Tier Design [04:13](https://www.youtube.com/watch?v=LqLoYksJ6do&t=253s)
Patch Pilot has two layers. The deterministic Go application handles orchestration: discovering vulnerable OCI artifacts, scanning images, and mapping which repository builds which image. This part is deliberately boring and simple. Inside it, agents are spawned for reasoning tasks—interpreting CI failures, distinguishing between real failures caused by previous changes versus flaky tests or infrastructure timeouts. The split between deterministic and agentic layers is the foundation of the security model.

### End-to-End Flow [05:13](https://www.youtube.com/watch?v=LqLoYksJ6do&t=313s)
The process starts with an assessment manifest (JSON listing images and their CVEs). The controller clones the target repository, creates a working directory with a ~2,000-word prompt and context, and invokes the CVE remediation agent. This agent performs the smallest effective change set—only bumping to versions that fix the specific CVE, not upgrading to latest. It verifies its own work by building the Dockerfile and rescanning the image. Critically, the agent only modifies files on the filesystem; it does not commit, push, or create PRs. The deterministic controller then vets changes for nonsense (empty files, stray binaries), commits, pushes, opens a PR, and watches CI. If CI fails, a PR remediation agent gets CI logs and workflow metadata to apply minimal fixes, with guardrails against reverting previous changes. This loops up to a maximum retry count before escalating to a human.

### Agent Retrospectives and Observability [08:19](https://www.youtube.com/watch?v=LqLoYksJ6do&t=499s)
After every agent invocation, the agent does a short retrospective: what went well, what went wrong, what tools are missing, what context would help next time. This addresses the open challenge of agent observability. Aggregating retros across PRs reveals two categories of issues: infrastructure problems (network failures, insufficient permissions) and repository complexity problems requiring system prompt modifications or repository-specific instructions.

### Production Screenshot and Diff [10:22](https://www.youtube.com/watch?v=LqLoYksJ6do&t=622s)
Johner shows a redacted production screenshot where the agent bumped a Go base image line and updated libcrypto3 and libssl3. The diff reveals the agent removed two previously pinned packages because they were already fixed by the base image bump—something Johner admits he would have missed himself, calling the agent "a good engineer."

### Credential Split and Blast Radius [11:24](https://www.youtube.com/watch?v=LqLoYksJ6do&t=684s)
Patch Pilot receives GitHub read/write access, OCI registry credentials, runtimes (Go, Python), linters, bash, and network access. However, the dangerous capabilities—GitHub write access and CI triggering—live only in the deterministic layer, never with the agent. This architectural choice limits the blast radius if the agent gets prompt-injected. The attack surface is wide: a small PR can involve 70,000 lines of changed code from dependency bumps.

### Prompt Injection Mitigation [12:57](https://www.youtube.com/watch?v=LqLoYksJ6do&t=777s)
Prompt injection can't be solved, only mitigated. Form3 uses prompt steering—telling the agent which directories contain untrusted data (vendor directories, CI logs). They also built end-to-end eval repositories with crafted injection attempts: deprecated functions that try to recruit the agent into malicious behavior, migration guides pointing to malicious code. Unknown injection vectors remain, so they don't rely solely on hope.

### The Docker Socket Problem [14:30](https://www.youtube.com/watch?v=LqLoYksJ6do&t=870s)
Sandboxing looks simple on a slide but breaks down when the agent needs to verify its work by building or running Docker containers. Giving the agent a Docker socket is essentially game over: it can spawn privileged containers, escape them, read environment variables and memory of other processes, and plant SSH keys. Form3 ran this way in production briefly but moved away from it. Johner evaluated Linux sandboxing technologies (landlock, bubble wrap, seccomp, unotify, fence, Kaniko, buildkit) but none compose well with containers or can contain a Docker daemon.

### Firecracker MicroVM Solution [15:31](https://www.youtube.com/watch?v=LqLoYksJ6do&t=931s)
The current design uses Firecracker microVMs for proper kernel-level isolation. The agent and Docker socket both live inside the microVM with its own kernel, solving the daemon escape problem. Network policies are applied separately for the deterministic and agentic layers. Since the agent's network needs vary by language ecosystem (Java vs. Python vs. Go), a DNS and TCP forwarder inside the microVM routes all traffic through a Vsock to a host process that enforces hostname, port, and CIDR-based rules. For advanced scenarios, a custom CA could enable TLS man-in-the-middle inspection.

### Assessment of Existing Sandboxing Tools [17:37](https://www.youtube.com/watch?v=LqLoYksJ6do&t=1057s)
Johner states that built-in sandboxes from Codex and Claude are worthless when Docker socket access is involved. Open Code and Pi lack sandboxes entirely. The gap isn't that tools don't exist—most are just in beta. He highlights microsandbox (YC-funded, open source) as his preferred choice if rebuilding today, noting it includes network access controls out of the box. Enterprise-grade features and traction are still maturing. He also points to the Kubernetes agent sandbox SIG and open sandbox projects as ongoing community efforts.

### Core Message and Community Invitation [20:40](https://www.youtube.com/watch?v=LqLoYksJ6do&t=1240s)
The key takeaway: the blast radius of an agent is an architecture decision. Keeping dangerous credentials in the deterministic layer rather than the agent defines the security model. Johner closes by inviting the community to share how they run agents in production, noting a dedicated sandbox panel the following day.
