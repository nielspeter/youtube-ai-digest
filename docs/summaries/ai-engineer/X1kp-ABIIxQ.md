## TL;DR
Dan Farrelly, CTO and co-founder of Inngest, argues that agent architectures have a short half-life because teams couple volatile components like prompts, models, and tools with stable execution logic. He advocates for a three-layer mental model—execution, context, and compute—and urges engineers to invest in a durable, decoupled execution layer that can survive rapid changes in the surrounding AI ecosystem.

## Key Takeaways
- Agent architectures degrade quickly because teams tightly couple components with vastly different lifespans.
- There are three conceptual layers to an agent system: execution (the brain), context (the knowledge), and compute (the hands).
- Different layers have different half-lives: prompts last weeks, models last months, but a well-designed execution layer can last years.
- Most current frameworks and custom-built systems merge orchestration with prompts and state, making it nearly impossible to swap out components without rewriting everything.
- A proper execution layer must provide resumability, durable and external state, flexible invocation patterns (crons, events, APIs, human-in-the-loop), and full-session observability.
- Sandboxes are ephemeral and stateless by design; using them for durability or state is an anti-pattern.
- Emerging agent architectures like background agents and autonomous loops are long-running, asynchronous, and require robust execution infrastructure to function.
- The execution layer sits between user input and system actions, making it the ideal hub for observability and outcome-based scoring.
- Inngest is positioned as a durable execution layer that lets teams plug in any model, framework, tool, or sandbox.

## Detailed Breakdown

### Introduction and the Problem with Agent Architectures [00:06](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=6s)
Dan Farrelly introduces himself as CTO and co-founder of Inngest, a system for reliably executing agents, workflows, and pipelines. He opens with the observation that anyone who has been building agents for more than six months has likely rewritten their architecture at least once. New models, frameworks, tool-calling standards, and patterns constantly emerge, rendering previous architectures obsolete. He challenges the audience to consider whether the code they shipped six months ago survived by accident or by design.

### The Three Conceptual Layers [02:14](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=134s)
Dan proposes a mental model of three discrete layers in an agent system. The **execution layer** is the brain, handling flow, state, durability, and retries. The **context layer** is the knowledge, encompassing models, prompts, tools, and memory—and it is the layer that changes the most. The **compute layer** is the hands, consisting of sandboxes, runtimes, and browsers. He emphasizes that these layers have different half-lives: prompts last weeks, models last months, but execution can last years if designed correctly. The core problem is that most teams couple everything together, causing one layer's short half-life to drag the others down.

### Issues with Current Approaches [04:21](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=261s)
Dan critiques common approaches: using frameworks that feel magical but hide abstractions, grabbing pre-built harnesses from frontier labs, or custom-rolling entire systems. In all these cases, abstractions are either missing, too high-level, or merged together. Orchestration gets buried inside chains or frameworks, state gets trapped in sandboxes, and retries get tangled with prompt logic. The result is that you cannot swap out any component without rewriting almost everything. He argues that teams must embrace change and invest in the stable execution layer.

### Defining the Execution Layer [05:53](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=353s)
The execution layer is defined as the system responsible for running code reliably and managing how, when, and whether each piece of work completes, independent of underlying infrastructure. It manages the full lifecycle: planning, calling models, running code, invoking sub-agents, looping, retrying, and coordinating. You should be able to swap models, context, and sandboxes while the execution layer remains unchanged.

### Requirements: Resumability and Durable State [06:56](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=416s)
The execution layer must enable resumability—picking up after failures without restarting from the beginning. For long-running agents (e.g., a 3-hour run), state cannot live in memory or on disk; it must be durable and external. Without this, teams resort to manual checkpointing or log-based hydration, and these abstractions start leaking into other layers, making the system harder to change.

### Requirements: Flexible Invocation Patterns [08:29](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=509s)
The execution layer must support a variety of invocation patterns: crons, event triggers, APIs, human-in-the-loop, sub-agents, and dynamic workflows. It must handle synchronous, asynchronous, and delayed invocations. Without flexible execution primitives, harness logic absorbs concepts like queues, workers, polling, and backoff, resulting in poor abstractions.

### Requirements: Full-Session Observability [09:01](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=541s)
Observability must span the entire session—not just LLM and tool calls, but database errors, permissions issues, triggers, and performance. A full trace from trigger through the entire stack is essential for debugging and improving agents over time.

### Sandboxes as Compute, Not State [09:33](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=573s)
Dan addresses the popularity of sandboxes. While agents need them for code execution, browsing, and file manipulation, sandboxes are ephemeral and stateless by design. Using them for durability or state snapshots is an anti-pattern. The execution layer should give the sandbox its context, sequence, and durability—execution is the brain, the sandbox is the hands.

### Emerging Architectures and Their Demands [10:35](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=635s)
Looking ahead, Dan identifies emerging patterns: background agents, dynamic workflows, autonomous loops, and agent factories. These are all long-running, asynchronous, and delegated. They require deep observability and must be inspectable by both humans and agents. Frameworks from even three months ago were not designed for these demands.

### Example: Background Agents [11:38](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=698s)
Background agents are not request-response systems; they may run for minutes or hours with hundreds of tool calls, guaranteeing at least one failure. Without proper infrastructure and observability, you cannot even debug an asynchronously running background agent.

### Example: Loop Architectures [12:11](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=731s)
Loops run continuously or on a schedule, assessing system state against goals and determining next steps. They require crons, sub-agent delegation, inspectable history, and reliability. Dan walks through a code example: a cron-based health check that runs every 30 minutes, invokes an LLM to assess system health, and triggers a triage agent if needed. The triage agent pulls detailed metrics, spins up a sandbox, clones code, and analyzes commits. A reviewer function runs weekly to evaluate the triage system's performance—checking logs, executed workflows, and sub-agent fan-outs—and suggests improvements.

### The Execution Layer as a Hub for Scoring [15:49](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=949s)
Because the execution layer sits between user input and system actions, it is an ideal place for observability and scoring. User feedback, actions, and session results all flow through it. Dan suggests deferring scoring after execution, using full trace information including inputs and outputs. Outcome-based scores—such as whether a triage resulted in an opened PR, or whether research was saved—are easier to implement when the execution layer can connect all these pieces.

### Inngest and Closing Remarks [16:19](https://www.youtube.com/watch?v=X1kp-ABIIxQ&t=979s)
Dan positions Inngest as the durable execution layer for AI agents. It provides durable steps, small primitives, event triggers, scheduling, agent-to-agent coordination, full session traces, and no infrastructure to manage. Teams can plug in any context layer (models, frameworks, tools) and any compute layer (sandboxes, browsers). He closes by urging the audience to build their harness with layered thinking, get the execution layer right, and embrace the fast pace of change.
