## TL;DR
Anant Shankhdhar, an AI engineer at Trisca (Risa Labs), presents a multi-agent system designed to automate oncology prior authorization workflows end-to-end with minimal human intervention. By leveraging deterministic decision engines, multi-source evidence reconciliation, and a clinical reasoning graph, the system confidently processes and submits eligible drug orders without human touch, escalating only complex or ambiguous cases to clinicians.

## Key Takeaways
- The oncology prior authorization workflow involves order intake, eligibility and benefits verification (EVBV), determining drug authorization status, and submission to payers.
- Four AI agents power the automation: EV Agent (eligibility), Auth Agent (authorization status), Necessity Agent (clinical reasoning), and Submission Agent (final filing).
- A unified service connects to various payer portals and APIs, normalizing output and using a deterministic engine to flag ineligible cases upfront.
- LLM-driven config generation and a self-healing loop reduce development time and mitigate runtime failures in RPA integrations.
- Single-source LLM extraction was insufficient due to sparse notes and non-deterministic outputs; multi-source evidence (authorization letters, payer rule knowledge bases) was required to boost confidence.
- A payer rule knowledge base built from portal checks, LLM extractions, and historical data identifies drugs that don't require authorization (NAR cases).
- The Medical Necessity Agent handles clinical reasoning by querying a patient medical graph of biomarkers against policy criteria, attaching confidence scores to escalate only uncertain cases.
- Orders where drugs are already authorized or don't require authorization can be fully eliminated from human review.
- The agents are being generalized beyond prior authorizations to support multiple oncology workflows.
- The "no touch" share of orders is growing through a combination of deterministic checks, multi-source evidence, self-healing RPA, and a clinical reasoning layer.

## Detailed Breakdown

### Introduction and Workflow Overview [00:00](https://www.youtube.com/watch?v=_cVfz88_j7A&t=0s)
Anant introduces himself as an AI engineer at Trisca and outlines the oncology prior authorization workflow. The process begins with daily order intake, followed by eligibility and benefits verification to check insurance coverage. Drugs are then categorized into three pathways: NAR (no auth required), auth on file (already authorized), and auth required (needs authorization). Although bots handle these steps, human review is traditionally required before submission. Anant was tasked with enabling some orders to proceed without any human touch, making confidence the key metric.

### Multi-Agent Architecture [02:06](https://www.youtube.com/watch?v=_cVfz88_j7A&t=126s)
To achieve no-touch automation, the system uses four agents: the EV Agent (eligibility and benefits verification), the Auth Agent (determining drug authorization status), the Necessity Agent (clinical reasoning for auth-required drugs), and the Submission Agent (final filing). Each agent handles a specific stage, and together they form an end-to-end pipeline.

### Eligibility and Benefits Verification (EV Agent) [02:38](https://www.youtube.com/watch?v=_cVfz88_j7A&t=158s)
The first challenge is that insurance data is scattered across dozens of portals, APIs, and documents. A unified service connects to different payer sources and returns normalized output. A coverage orchestrator routes each patient to either an API or RPA path, and the result is passed to a deterministic engine that checks if coverage is active. To avoid building custom integrations for every portal, the team created a repository of custom and popular RPA actions, used LLMs to generate portal configs, and implemented a self-healing loop to handle runtime failures during production hours.

### Authorization Status Extraction (Auth Agent) [06:17](https://www.youtube.com/watch?v=_cVfz88_j7A&t=377s)
For drugs that are already authorized (auth on file) or don't require authorization (NAR), human oversight can be removed. An initial LLM extraction pipeline categorized drugs from patient notes, but this approach had issues: notes often lacked sufficient data, and LLM extraction is non-deterministic, meaning outputs couldn't be blindly trusted. To solve this, the team added two additional evidence sources: historical authorization letters (proving prior approval) and a payer rule knowledge base (a SQL database built from portal checks and LLM extractions identifying NAR drugs on a monthly basis). By reconciling multiple sources, the system extracts auth statuses with higher confidence and can fully eliminate certain orders from human review.

### Payer Rule Knowledge Base for NAR Cases [09:59](https://www.youtube.com/watch?v=_cVfz88_j7A&t=599s)
Anant explains how the payer rule knowledge base is built. Documents containing time-based NAR information are processed through configurable LLM extraction to produce drug-wise constraints per payer. Historical information about how organizations treat specific drugs is also stored. Regular portal checks supplement this data. This deterministic layer, combined with eligibility verification, allows the system to flag non-verifiable cases early and move confident cases forward.

### Medical Necessity Agent and Clinical Reasoning [11:34](https://www.youtube.com/watch?v=_cVfz88_j7A&t=694s)
For auth-required drugs, clinical reasoning is necessary. The Medical Necessity Agent reads patient notes, policy criteria, and queries a patient medical graph—a graph of extracted biomarkers. The agent uses an LLM to answer clinical questions with supportive and contradictory facts, attaching a confidence score. High-confidence answers proceed automatically; cases with insufficient information are escalated to a clinician. Anant notes the team has a publication detailing this work.

### Submission Agent and End-to-End Flow [14:11](https://www.youtube.com/watch?v=_cVfz88_j7A&t=851s)
After medical necessity validation, all assembled information is submitted back to payers via the Submission Agent, which functions similarly to the EV Agent with customized, LLM-driven integrations for each payer. Anant presents the full workflow graph, highlighting the "green" path where orders move from eligibility verification through to submission without human touch. He also notes that these agents are being generalized for use across multiple oncology workflows beyond prior authorizations.

### Conclusion and Key Strategies [16:14](https://www.youtube.com/watch?v=_cVfz88_j7A&t=974s)
Anant summarizes the approach: starting with deterministic checks, using agents only where rules can't decide, leveraging multi-source evidence to outperform single-source extraction, adding self-healing for scalable RPA, and introducing a reasoning layer for genuine authorization cases. The no-touch share of orders continues to grow.
