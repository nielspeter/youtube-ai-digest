## TL;DR
Dmitry Petrov explains why coding agents fail on physical, unstructured data (video, sensor, robotics telemetry) and introduces the concept of a "data harness"—a system layer that lets agents see, process, test, and remember this data efficiently. He demonstrates how DataChain, an open-source project, uses Pydantic schemas, a distributed execution engine, and a knowledge base to give agents the context they need, mirroring approaches Anthropic and OpenAI have described for structured data but adapted for the messier physical data world.

## Key Takeaways
- Agents struggle with unstructured/physical data because they lack the "harness" needed to understand, process, and remember binary files like video and sensor recordings.
- Anthropic reported only 21% accuracy for data agents without specific data harnesses; OpenAI described needing six layers of context—both focused on structured data, leaving physical data an unsolved frontier.
- A small number of video files can explode into millions of nested objects (frames, detections, bounding boxes), making naive JSON-on-S3 approaches unscalable.
- Pydantic data models let researchers define schemas in pure Python, avoiding a separate SQL stack and making metadata queryable in a local database.
- A proper execution engine for unstructured data must support parallel/distributed compute, incremental updates, and checkpointing so expensive LLM processing isn't wasted on failures.
- Instead of running slow Python scripts over raw binaries, agents should build reusable metadata layers (star schemas, one-big-table approaches) that answer analytical questions in fast SQL-like queries.
- A knowledge base of Markdown files—containing dataset descriptions, session context, schemas, stats, and source code—creates data lineage and lets teammates and agents reuse prior compute instead of reprocessing.
- The full stack combines object storage, a compute engine, a dataset DB, and a knowledge base; without this harness, frontier models' intuition leads them in the wrong direction on physical data.

## Detailed Breakdown

### The Problem: Agents and Physical Data [00:02](https://www.youtube.com/watch?v=bUJgirn4_yc&t=2s)
Petrov opens by noting that agents perform poorly on physical data—video, sensor data, robot telemetry—because they lack the supporting infrastructure. He cites recent findings from Anthropic (21% accuracy without data harnesses) and OpenAI (six layers of context needed for data agents), but observes these address structured business data, not the messy unstructured binaries he works with.

### Background and the "Neutron Star" Metaphor [01:35](https://www.youtube.com/watch?v=bUJgirn4_yc&t=95s)
With a decade in data—including building DVC (Git for data) and now DataChain—Petrov explains that agents need more than an LLM brain; they need a harness to see, touch, verify, and remember data. He introduces the "neutron star" metaphor: a handful of video files can look small but contain millions of nested objects (clips, frames, detections, bounding boxes), creating an explosion of metadata that overwhelms naive approaches.

### Schema Management with Pydantic [03:09](https://www.youtube.com/watch?v=bUJgirn4_yc&t=189s)
He critiques common practices—dumping metadata to millions of JSON files on S3 (slow, inconsistent) or centralizing in a database (introduces a separate SQL stack). DataChain's approach uses Pydantic models in pure Python so schemas, code, and queries share one language, transpiling to SQL under the hood. This eliminates the "SQL island" problem and makes complex nested objects queryable.

### Live Demo: Dashcam Video Analysis [05:43](https://www.youtube.com/watch?v=bUJgirn4_yc&t=343s)
Petrov demonstrates DataChain with a dashcam video dataset. After installing the tool and a skill for Claude Code, the harness asks clarifying questions (model choice, YOLO size, velocity tracking method, granularity, dataset scope). It processes 91 clips in 24 minutes, generating ~100,000 records. He then queries how many videos contain people (82 of 91) and shows the generated Python code running a fast database query with a simple `label == "person"` filter.

### The Generated Data Model [11:32](https://www.youtube.com/watch?v=bUJgirn4_yc&t=692s)
He walks through the agent-generated Pydantic data model: fields include `frame_id`, `timestamp`, `class_id`, `confidence`, a nested `bounding_box`, and a `file` object with path, checksums, e-tag, and size. This metadata becomes a database row instead of a scattered JSON file, enabling instant analytical queries.

### Execution Engine for Unstructured Data [13:39](https://www.youtube.com/watch?v=bUJgirn4_yc&t=819s)
For terabytes of messy data, the harness needs an execution engine. Petrov favors a Dask-like model connecting Python functions, Pydantic schemas, and storage. Developers write plain Python functions with typed inputs/outputs; the engine handles parallelization across machines. He notes you can simply specify resource needs (e.g., "40 machines") and the engine distributes work based on files and schemas.

### Incremental Updates and Checkpointing [18:18](https://www.youtube.com/watch?v=bUJgirn4_yc&t=1098s)
Because LLM processing of binary files is expensive and time-consuming, losing compute to mid-run failures is a major pain. Petrov insists incremental updates and checkpointing are must-haves: fix a bug, rerun, and the system catches up without recomputing finished work. New files added to a bucket trigger processing of only the new data.

### Fast Testing via Metadata Layers [19:19](https://www.youtube.com/watch?v=bUJgirn4_yc&t=1159s)
Testing in data projects is even more critical than in software because there's usually only one correct answer. Running Python scripts over raw binaries is the slowest, most expensive approach. Instead, agents should build reusable metadata layers—borrowing dimensional modeling, star schemas, and one-big-table techniques from the data warehousing world—so questions are answered via fast SQL-like queries. The agent proactively builds these layers to be general enough for related questions, not just the one asked.

### Knowledge Base and Memory [21:58](https://www.youtube.com/watch?v=bUJgirn4_yc&t=1318s)
Petrov describes how DataChain stores dataset context as Markdown files: description, session context (why it was built), storage dependencies, data preview, schema, stats, and source code. This creates data lineage connecting raw files, code, and warehouse results. When shared, teammates' agents reuse prior compute instead of reprocessing, eliminating redundant costs.

### The Full Stack [25:40](https://www.youtube.com/watch?v=bUJgirn4_yc&t=1540s)
Petrov assembles the pieces: unstructured data in object storage at the bottom, a compute engine extracting metadata, a dataset DB organizing slices, and a knowledge base for sharing. He stresses that standard coding agents (Copilot, Codex, Claude Code) fail here not because of model weakness but because the "laws of physics change" for physical data. The solution isn't a stronger model but a data harness that understands those laws—implemented in the open-source DataChain project.
