## TL;DR
Mike Phipps from the Gates Foundation explains how their team built the Strategic Intelligence Platform (SIP), an enterprise-wide knowledge graph that structures operational data for agentic retrieval. The core thesis is that as AI models and chat interfaces become commoditized, the durable competitive advantage lies in deeply modeling your organization's internal processes, tacit knowledge, and data relationships—not in the UI or the underlying LLM.

## Key Takeaways
- **Your moat is your data model.** As AI capabilities commoditize, defensible value comes from modeling tacit organizational knowledge—procedural understanding that AI needs to answer questions correctly.
- **The Gates Foundation's scale is massive.** Over 4,000 employees, 2,000+ grants per year, 100+ countries, and $7B+ in annual disbursements—all of which must be structured for intelligent retrieval.
- **SIP is a cross-system semantic graph layer.** It unifies siloed systems of record into a single knowledge graph that agents can traverse at query time.
- **Engaging data owners is critical.** Understanding field meanings, join logic, reporting conventions, and data limitations requires deep collaboration with the people who own the data.
- **Multiple hierarchies coexist in the graph.** The model accommodates funding hierarchies (additive DAG), management hierarchies (with direct/indirect rollups), and people/org hierarchies simultaneously.
- **Governance becomes more acute with AI.** PII masking, sensitive data classification, and per-user entitlements must be carefully managed because AI makes previously inaccessible data far more accessible.
- **The chat interface is not your moat.** Users already have ChatGPT and Claude; the Foundation serves its platform where users already are, via MCP, rather than competing on UI.
- **Evals reveal gaps in your data model.** Targeted eval questions matched to reporting standards, with live graph queries compared against agent outputs, create a feedback loop that improves the model over time.
- **Neo4j is the graph database powering the platform**, with modified MCP servers connecting the graph to agent experiences.

## Detailed Breakdown

### The Defensibility Question [00:01](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=1s)
Phipps opens by framing the core problem: with AI moving fast and tools like Claude's code generation enabling rapid builds, what's actually defensible once you push to production? He raises practical concerns—monitoring, upkeep, dependencies—and questions whether another chat app provides meaningful product differentiation over existing SaaS products. His team concluded that their competitive advantage is their understanding of internal processes and tacit knowledge, which remains durable regardless of new model releases.

### Gates Foundation Context and Scale [02:36](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=156s)
To explain what they're modeling, Phipps gives background on the Gates Foundation's 25-year scope: child mortality, nutrition, agriculture, education, and more. The foundation's work generates enormous structured data—over 2,000 grants in 2023 alone, many at $5M+, across 100+ countries, with $7B+ in annual disbursements and 4,000 employees. Multiple divisions (Global Development, Global Health, Gender Equality, USP) fund work through different organizational structures, which the data model must reflect.

### Strategic Intelligence Platform (SIP) Overview [05:15](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=315s)
SIP is described as structuring operational data for agentic retrieval. The architecture includes siloed systems of record (structured and unstructured), consolidated into an internal data lakehouse. A data curation layer processes the data, and SIP serves as a cross-system semantic graph layer exposed to agents via MCP, with agentic chat and agentic workflows as the user experience.

### Data Curation and Governance [06:18](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=378s)
Phipps emphasizes that engaging data owners is critical for understanding tacit knowledge: field meanings, join logic, reporting conventions, and data limitations. The curation pipeline handles pre-processing, filtering, deduplication, semantic chunking, figure-to-text conversion, tagging, and metadata creation. Governance is a major concern—AI makes data more accessible, expanding the risk surface. PII must be masked, sensitive data classified, and entitlements enforced per user.

### The Data Model: Hierarchies and Relationships [08:50](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=530s)
The graph model is built around multiple hierarchies. The first is an additive DAG with five levels representing the funding path: funds → portfolios → investments, with multiple funding teams supporting individual investments. The second hierarchy is management-based, where each level matters independently, and derived edges (like "rollup manages") are precomputed to handle direct and indirect team management attribution. A third hierarchy models people—org charts, owners, attendees, directors—stitching HR data into the broader organizational picture.

### Connecting Siloed Systems and Documents [13:04](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=784s)
The graph's power comes from finding common entities across siloed source systems and stitching them together, enabling agents to traverse and understand the full organizational structure. Documents add the unstructured layer: meetings have documents, which have semantic chunks with full-text indexes for retrieval. These connect back to the main organizational structure, combining structured and unstructured data in one graph.

### Serving Users via MCP [15:40](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=940s)
Phipps argues the chat interface and general agent interaction are not defensible—users already have ChatGPT and Claude. So SIP is served where users already are, through MCP. The team forked Neo4j's off-the-shelf MCP servers, adding schema updates and state-passing (conversation IDs, message numbers). Beyond general chat, they're building more constrained workflow experiences through sandbox-based agents that pull from the same knowledge graph backend.

### Evals as a Data Model Feedback Loop [17:12](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=1032s)
Evals reveal ambiguities and gaps in the data model. The team works with data owners to create targeted eval questions matching reporting standards, separated into complexity tiers. Because structured data constantly changes, they pull from the live graph at eval runtime and compare agent outputs against expected graph query results. An LLM-as-judge measures pass@1 and stability (consistent answers to repeated questions). Gaps found through evals feed back into updates to the data model, domain rules, and schema descriptions. Remaining misses tend to be genuinely ambiguous questions where the answer isn't wrong but doesn't match user intent.

### What's Ahead for SIP [19:20](https://www.youtube.com/watch?v=jt1Pbr_n6oU&t=1160s)
Future work includes continuing to fill out existing systems-of-record data, expanding the primary graph to additional enterprise-wide datasets, exploring a federated graph experience (where specific teams link their own data to the main enterprise graph), and building additional constrained agentic experiences.
