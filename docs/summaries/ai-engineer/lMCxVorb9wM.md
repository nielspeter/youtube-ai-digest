# You Didn't Ship a Bug. You Just Wrote It for a Human. - Ravi Madabhushi, Scalekit

## TL;DR
Ravi Madabhushi argues that traditional authentication and authorization architectures, built for human users and deterministic programs, break down when applied to non-deterministic AI agents. He calls for rethinking access controls with fine-grained, context-aware, time-bound permissions and full audit visibility to ensure agents operate with least privilege and deterministic guardrails.

## Key Takeaways
- Systems designed for humans and deterministic APIs don't scale safely for non-deterministic agents.
- Agents often receive far broader permissions and scopes than their actual responsibilities require.
- Traditional auth models assume the authenticating entity is the same as the acting entity; agents break this by acting on behalf of users.
- Unlike traditional programs, agents are probabilistic—you can't inspect code to guarantee what they will or won't do.
- Most MCP servers today surface all available tools to agents regardless of which user authorized them, creating security risks.
- Agents need their own identities, bound to principals, with extremely fine-grained, attribute-level, context-level scoping.
- Permissions should be time-bound and goal-specific, not broad and persistent.
- Agents should default to least privilege and be able to request just-in-time authorization for elevated scopes.
- Full audit visibility—who acted, on behalf of whom, with what authorization, for how long—is essential.
- OAuth is a good starting point but insufficient on its own; something beyond OAuth is needed for agent-specific access control.

## Detailed Breakdown

### The Latency Spike That Started It All [00:00](https://www.youtube.com/watch?v=lMCxVorb9wM&t=0s)
Ravi opens with a story about noticing rhythmic latency spikes every 15 minutes in Scale Grid's identity platform. The root cause was a "last seen" timestamp updated for every user action. When agents began hitting their APIs, this timestamp was being updated 60 times faster than with human users, creating unnecessary database write pressure. The fix was simple—batching updates at the second level—but it triggered a deeper investigation into how agent behavior fundamentally differs from human behavior.

### Background in Identity and Authentication [01:34](https://www.youtube.com/watch?v=lMCxVorb9wM&t=94s)
Ravi shares his 10-year background building identity platforms at Freshworks, serving millions of daily human users and APIs. He explains that APIs accessed by machines were still manageable because those machines ran deterministic programs written by humans. However, the landscape has shifted dramatically in the last 3–4 years with the rise of agents.

### The Over-Permissioning Problem [02:37](https://www.youtube.com/watch?v=lMCxVorb9wM&t=157s)
Through Scale Grid's ringside view of how developers build agents, Ravi observes that most agents have far more permissions and scopes than their jobs require. This isn't due to developer carelessness but because existing primitives don't support extremely fine-grained permissions for agents—it became a default pattern.

### Two Existing Authentication Slots [03:09](https://www.youtube.com/watch?v=lMCxVorb9wM&t=189s)
Ravi outlines the two predominant auth models: (1) a human authenticates and acts through their own application or script using their API key, and (2) a service account is created with its own identity and permissions (the SPIFFE/OAuth model). Both models share the philosophy that whoever authenticates is the one acting, with permissions fixed at registration time.

### Why Deterministic Programs Were Safe [04:44](https://www.youtube.com/watch?v=lMCxVorb9wM&t=284s)
Traditional programs—whether using passwords, API keys, web sessions, or service accounts—behave deterministically. Developers wrote them, and you could inspect the code to verify behavior. Even Google's developer account security review process involves examining codebases for appropriate practices. These programs always stay in their lane and can't do what they weren't programmed to do.

### How Agents Break the Fundamental Assumption [06:18](https://www.youtube.com/watch?v=lMCxVorb9wM&t=378s)
Agents break two core assumptions. First, the principal (the user) is not the same as the actor (the agent)—agents need delegated access to act on behalf of users, accessing their Gmail, Salesforce, etc. Many systems don't even support OAuth properly, so there's no "on behalf of" principal working. Second, agents are non-deterministic—just because an agent does something today doesn't guarantee it will behave the same way tomorrow or even on the next run.

### The MCP Server Problem [08:22](https://www.youtube.com/watch?v=lMCxVorb9wM&t=502s)
Ravi highlights that most MCP servers they've worked with don't limit tool context access based on which user authorized the agent. They typically surface all tools the user has access to or all tools the application supports, letting the agent determine what it can or cannot do. This leads to agents picking wrong tools and doing wrong things, with only runtime checks in the application as a backstop.

### What Needs to Change: Fine-Grained Agent Credentials [09:23](https://www.youtube.com/watch?v=lMCxVorb9wM&t=563s)
Ravi calls for agents to have their own identities and extremely fine-grained credentials—far beyond current OAuth scopes. He uses Gmail as an example: current scopes only ask whether a client can send emails on your behalf, but don't allow constraints like acting only at certain hours, reading emails only from specific senders, or sending only to specific recipients. Agents need attribute-level, context-level, and principal-level scoping.

### Least Privilege and Just-in-Time Authorization [10:55](https://www.youtube.com/watch?v=lMCxVorb9wM&t=655s)
Agents should be least privileged by default and able to request just-in-time authorization for elevated scopes. Every agent has a goal and a job, so access should be deterministically limited to only the tools and jobs it needs. Ravi emphasizes this is a present-day problem, citing incidents of agents deleting production databases.

### Real-World Example: ref.tools [11:26](https://www.youtube.com/watch?v=lMCxVorb9wM&t=686s)
Ravi mentions a customer, ref.tools, whose product gives context to coding agents. They don't even have humans as actors and built their entire OAuth scoping system to handle agent access correctly—demonstrating this is a problem of today, not tomorrow.

### Visibility and Deterministic Guardrails [11:59](https://www.youtube.com/watch?v=lMCxVorb9wM&t=719s)
Ravi stresses the need for absolute visibility into every agent action: who took it, on behalf of whom, who authorized it, when, what authorization was given, and for how long. Without this visibility and deterministic control, you're simply praying the agent doesn't do something harmful—and "praying is not a strategy." He closes by urging developers to rethink architectures built for humans and APIs, starting with OAuth but going beyond it for fine-grained agent access control.
