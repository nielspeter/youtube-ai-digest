# Agent Output Is Not UX: Rendering Layer Your LLM Pipeline Is Missing - Bala Ramdoss, Amazon Lens

[![Agent Output Is Not UX: Rendering Layer Your LLM Pipeline Is Missing - Bala Ramdoss, Amazon Lens](https://i.ytimg.com/vi/maTp79FD9gI/hqdefault.jpg)](https://www.youtube.com/watch?v=maTp79FD9gI)

## TL;DR
Bala Ramdoss draws on years of building Amazon Lens to argue that the real bottleneck in shipping agentic AI products isn't the model—it's the rendering layer between model output and what the user sees. He outlines three practical patterns—versioned rendering contracts, streaming into typed components, and a backend-for-frontend (BFF)—that let teams turn raw LLM intent into safe, native, snappy mobile UX.

## Key Takeaways
- Agent output is not UX; the model does the hard work, but the delivery layer between model and screen determines whether the product succeeds.
- "Generative UI" now has a name and an open spec (A2UI from Google); instead of raw text or HTML, the agent describes UI as data the client renders with native widgets.
- Generative UI is a spectrum: control (pre-built components), declarative (compose from a catalog), and fully open-ended. Most production mobile apps live in the bottom two tiers.
- Mobile clients cannot be meaningfully patched after release—if a client meets an unknown content type it crashes, sometimes for weeks in un-updated installs.
- Pattern 1 — Rendering Contract: make the model aware of client capabilities and version maps; the model picks from a fixed menu of components and never invents new ones.
- Pattern 2 — Streaming: stop chasing total latency and start chasing "time to first chunk"; show skeletons, partial fills, or thinking states to keep users engaged.
- Pattern 3 — BFF (Backend for Frontend): the BFF absorbs model output, applies platform-specific rules, hydrates components with action payloads, and carries conversational context across turns.
- You can reuse existing production UI components (flight rows, product cards) rather than building a new "agentic look"—keeping the same brand, density, and familiar feel.
- AI users have moved past the forgiving phase; they now expect transparency about what the agent is doing.
- None of these patterns are about the model—the model was fine. The rendering layer is what ships the product.

## Detailed Breakdown

### The Reservation Example [00:01](https://www.youtube.com/watch?v=maTp79FD9gI&t=1s)
Ramdoss opens with a relatable scenario: asking an AI assistant to reserve a restaurant table. The model returns correct information—phone number, hours, even the walk-in oyster bar—but the user still has to do all the work of actually booking. He contrasts this with an ideal outcome: a date, a time, a couple of taps, and you're done. The agentic product already has the tools; the missing piece is the layer between the model and human interaction.

### Introduction and Background [01:05](https://www.youtube.com/watch?v=maTp79FD9gI&t=65s)
He introduces himself as Balaram Das, a builder of customer-facing apps for over a decade, with the past six years spent on Amazon Lens—Amazon's suite of camera features powered by AI that lets users shop via images, screenshots, and barcodes. He notes the talk comes from experience with products on millions of mobile devices and includes a disclaimer that the opinions are his own, not his employer's.

### The Delivery Problem [01:37](https://www.youtube.com/watch?v=maTp79FD9gI&t=97s)
When people think of agentic AI UX, ChatGPT comes to mind—cards, carousels, lists. But building this surfaces hard problems early: will the experience be snappy or slow? Do you get everything at once or incrementally? How do you scale across fragmented mobile versions and device capabilities? None of these problems stem from the model; they are delivery problems living between model output and the screen—the layer that decides product success.

### Naming the Layer: Generative UI and A2UI [03:10](https://www.youtube.com/watch?v=maTp79FD9gI&t=190s)
Two years ago, Ramdoss says, he had no name for this layer and solved it from scratch every time. Now it has a name—Generative UI—and an open spec from Google called A2UI. Instead of handing back raw text or HTML, the agent describes UI as data: a list of components the client renders with its own native widgets. He frames it as a spectrum laid out by Co-pilot Kit in three tiers: control (model picks a pre-built component), declarative (model composes from a catalog—where A2UI sits), and fully open-ended (model generates novel UI on the fly). Most production mobile apps stay in the bottom two tiers to remain safe.

### The Mobile Constraint [05:15](https://www.youtube.com/watch?v=maTp79FD9gI&t=315s)
On the web, a broken render can be fixed and live in minutes. Mobile apps, however, sit on hundreds of millions of installs with no control over when users update. If a client encounters an unknown content type, it doesn't gracefully degrade—it crashes, and keeps crashing for days or weeks for un-updated users. The one rule that holds for mobile clients: you cannot meaningfully patch the client.

### The Simplified Pipeline [05:46](https://www.youtube.com/watch?v=maTp79FD9gI&t=346s)
Zooming out, Ramdoss describes the pipeline: a version-aware context phase feeds the model (context engineering), the model outputs typed UI intent, which flows through a BFF (backend for frontend) to a client renderer that draws it and falls back safely when it cannot. He identifies three patterns to explore: the rendering contract, streaming, and the BFF.

### Pattern 1: The Rendering Contract [06:17](https://www.youtube.com/watch?v=maTp79FD9gI&t=377s)
The contract makes the model aware of client capabilities. You maintain a version map of capabilities—for example, a new flight card UI introduced in version 2.0 should only be surfaced to the model from that version onward. The model-facing half of the contract specifies that the model should stream blocks of UI components (a conversation block for text, a UI block for rendering) rather than plain text. It encodes layout rules: one to three flights render as a swipeable carousel, four or more as a vertical list. The model picks the intent but never invents a component—it chooses from a fixed menu you provide. As features scale to more surfaces, engineering the context so the model succeeds becomes the hard part.

### Pattern 2: Streaming [08:55](https://www.youtube.com/watch?v=maTp79FD9gI&t=535s)
Traditional API calls—make a request, wait for the full response—don't work well with LLMs because latency is higher and there are additional safety checks plus a new dynamic UI layer. Streaming solves this by rendering chunks incrementally: skeleton, partial fill, complete. This shifts what you measure: stop chasing total latency and start chasing time to first chunk—the first useful thing the user sees. The traditional loading spinner won't cut it for AI features. Ramdoss shares a product example from Amazon Lens Live, where users can tap on objects of interest while waiting for results. If you don't control the full screen, show a thinking state—but use it sparingly, because users have moved past the forgiving phase and expect to know what's happening. He cites Gemini's approach of giving a glimpse into agent activity, which builds trust even for ten-second tasks.

### Pattern 3: The BFF [11:31](https://www.youtube.com/watch?v=maTp79FD9gI&t=691s)
The BFF is the most important part. It embodies server-driven UI—a spectrum of server control. The BFF owns platform-specific rules (Android vs. iOS), lets the client make fewer decisions, and simply draw what's handed to it. Beyond layout, the BFF handles hydration and adds actions: every rendered element carries an action payload for taps, deep links, and even impression metrics. It carries conversational context across turns so the next response knows what came before. Critically, you can do this with existing apps—reusing the flight rows, product cards, and components already in production. You're not building a new agentic look; you're maintaining the same brand, density, and familiar feel so it looks and feels native.

### Takeaways [13:03](https://www.youtube.com/watch?v=maTp79FD9gI&t=783s)
Ramdoss summarizes three takeaways: (1) models are already capable—feed them a properly typed, versioned contract so they pick the right CX; (2) stream into typed components, not text, and let users know what the agent is doing; (3) let the BFF absorb model output so the client stays dumb and safe. He reiterates that none of these are about the model—the model was fine. This layer is what ships the product. He closes by returning to the opening example: your agent output is not the CX; you build on it.
