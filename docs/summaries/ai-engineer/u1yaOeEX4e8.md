## TL;DR
Ritvik Pandya of JP Morgan Chase explains how representing API request processing as short-lived Directed Acyclic Graphs (DAGs) enables precise anomaly detection and drift identification. By establishing baselines for each node in the execution graph, teams can localize problems, categorize drift types, and automate remediation—significantly reducing mean time to discovery in real-time payment systems.

## Key Takeaways
- Execution graphs are short-lived DAGs representing end-to-end request processing across all services, unlike persistent property graphs (e.g., Neo4j).
- Establishing a baseline for normal request execution allows for a fast "tier one" check; only deviations trigger deeper analysis.
- Anomalies are one-off incidents (e.g., a traffic jam), while drift is a gradual pattern change over time requiring a new baseline.
- Drift falls into three categories: structural (nodes added/removed), scale/volume-related, and co-variate (shifting request composition, like more international vs. local traffic).
- Graph-based approaches pinpoint exactly which node is causing delays, rather than just detecting that overall request time increased.
- OpenTelemetry feeds trace data asynchronously (often via Kafka) into the assessment system, supporting both hot (fast) and reconciliation (accurate) paths.
- Challenges include handling delayed telemetry events without triggering false structural-change alarms, managing cold starts for new endpoints, and tailoring baselines per client/use case rather than using generic thresholds.
- Automated remediation should be risk-assessed and rolled out incrementally (e.g., 5–10% of nodes first) before full deployment.
- Statistical detection methods like MMD (Maximum Mean Discrepancy) and KL divergence can be applied once baselines are confirmed.
- The approach reduced mean time to discovery to a single time window instead of requiring comparison across multiple windows.

## Detailed Breakdown

### Introduction and Concept of Execution Graphs [00:01](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=1s)
Ritvik Pandya introduces himself as leading the payments team at JP Morgan Chase. He distinguishes execution graphs from persistent/property graphs (like Neo4j): execution graphs are short-lived, representing how a single request is processed end-to-end. The goal is to holistically identify deviations in request processing and automate detection and remediation.

### Representing Request Processing as a DAG [01:35](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=95s)
A typical request flows through an edge/ingress layer, gateways, authentication/authorization, orchestration, and parallel service calls, culminating in client notification. Representing this as a Directed Acyclic Graph (DAG) clarifies service execution order, context passed between nodes, and handles retries and loops as separate trackable entities.

### Tiered Checking and Localizing Deviations [03:09](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=189s)
Pandya compares the system to airport security: a tier-one check compares the request against a known baseline. If everything looks good, no deeper analysis is needed. If a delay or deviation is found, the system checks for structural changes (nodes added or removed), then investigates scale deviations, divergence, or exponential issues. Baselines vary per client—for example, local clients vs. international clients have different expected thresholds.

### Anomaly vs. Drift: The Commute Analogy [07:21](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=441s)
An anomaly is a one-time incident—like an unexpected traffic jam making your commute longer one day. Drift is a gradual pattern change—like realizing over a year that your commute now takes 20 minutes longer than it used to. Drift requires establishing a new baseline rather than treating it as an incident.

### Three Categories of Drift [08:55](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=535s)
1. **Structural drift**: A new node is added or an existing one removed (e.g., a coffee shop adding a membership check step). Baselines and alerts must be updated.
2. **Scale/volume drift**: A service can no longer handle expected volume, requiring scale-up or asynchronous processing.
3. **Co-variate drift**: The composition of requests changes over time (e.g., shifting from 60% local / 40% international to a higher international ratio), requiring either separate baselines per category or an adjusted average.

### Benchmarking and Training the System [06:50](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=410s)
Pandya references benchmarking with OpenTelemetry and "star bench" (likely StarBench), where millions of traces over 7 days were injected, anomalies were deliberately introduced, and the system was trained before going live.

### System Architecture: Telemetry, Hot Path, and Reconciliation [12:39](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=759s)
OpenTelemetry continuously feeds data. Root cause analysis is performed once all data points are available. The system identifies the drift type, determines the action, assesses risk, and rolls out solutions incrementally (5–10% first, then 100%). Data is fed asynchronously via Kafka, supporting a hot path for fast automated decisions and a reconciliation path for more accurate, slower analysis.

### Example: Pinpointing Node-Level Delays [14:15](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=855s)
If overall request processing typically takes 700ms but suddenly spikes, the graph approach reveals exactly which node is slow (e.g., an FX rate service). Without the graph, you only know the total is delayed. In payments, speed is critical, so this localization is essential.

### Challenges: Delayed Events, Cold Starts, and False Alarms [15:18](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=918s)
If one of seven nodes delays sending telemetry, the system might falsely detect a structural change (seeing only six nodes). Tail-based systems help track when a service request starts and ends per node. Cold starts require new endpoints to get their own baselines rather than inheriting generic ones. Detection methods like MMD or KL divergence are used, and admin confirmation helps classify problems before automation.

### Results and Best Practices [17:22](https://www.youtube.com/watch?v=u1yaOeEX4e8&t=1042s)
Mean time to discovery dropped significantly—issues were identified within a single time window rather than multiple comparison windows. Best practices include: using well-tuned labels, creating specific baselines per use case (e.g., real-time payments vs. wire payments vs. generic POST requests), defining well-tuned windows per client, ensuring explainability of data (like a doctor explaining a health score), and making the system aware of new deployments to inform rollback decisions.
