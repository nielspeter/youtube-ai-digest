## TL;DR
ZS Associates built a multi-agent pipeline for pharma commercial analytics—mimicking how human analysts detect signals, find root causes, and recommend actions—but found the outputs incoherent because context was lost between agents and no single agent owned the end-to-end reasoning. By consolidating into a single agent, separating deterministic signal detection from agentic investigation, and building a knowledge graph as a control plane, they achieved in 20–30 minutes what previously took analysts 3–4 weeks.

## Key Takeaways
- **Don't impose human workflow structures on AI architecture.** Splitting work across multiple agents to mirror how analysts work led to context loss and incoherent outputs.
- **Separate deterministic steps from agentic reasoning.** Signal detection should use statistical methods and thresholds, not language models; agents should investigate, not identify signals.
- **One agent should own end-to-end reasoning.** Sub-agents can handle focused investigative tasks, but judgment and synthesis must remain centralized.
- **Knowledge graphs should serve as a control plane, not just a data lookup.** The graph dictates what paths the agent can take, what hypotheses it can evaluate, and keeps investigation bounded.
- **Context handoff between agents is fragile.** Even when individual agents derive correct facts, downstream agents may not understand the weight or significance of upstream findings.
- **Observing Claude in a raw coding environment informed the redesign.** Watching it repeatedly write functions and query databases inspired a lighter, more consolidated architecture.
- **Domain expertise is essential for building the knowledge graph.** Mapping entities, KPIs, and their relationships required sitting with experienced pharma commercial analytics experts.
- **The loop pattern works well with a graph.** The agent traverses graph edges as hypotheses, evaluates them against real data, and iterates until root cause is found or hypotheses are exhausted.
- **Performance gains were dramatic**—20–30 minutes of agent runtime replaced 3–4 weeks of analyst effort.

## Detailed Breakdown

### Introduction and Context [00:01](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=1s)
Subbiah Sethuraman (head of AI engineering) and Abhilash Asokan (director of AI engineering) introduce themselves from ZS Associates, a technology firm working with top global companies including major pharmaceutical firms. The talk focuses on their experience building multi-agent pipelines—the mistakes they made, what they learned, and how they fixed the problems.

### Pharma Commercial Domain Overview [00:32](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=32s)
Subbiah orients the audience to the pharma commercial domain, explaining that pharma has two main functions: R&D (drug discovery and clinical trials) and commercial (taking a drug to patients). Within commercial, key functions include brand performance across markets, field force effectiveness, and patient journey analysis. Analysts in this space follow four steps: signal detection (e.g., a drop in prescriptions), root cause analysis (why the signal is happening), action determination (what to do about it), and outlook assessment (will the action improve sales performance).

### The Multi-Agent System They Built [02:07](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=127s)
To mimic analyst behavior in an agentic way, the team built separate agents for each step: a signal detection agent, a source localization agent (to identify where a problem is concentrated—region, payer, etc.), a driver attribution agent (to find the real reason), and a synthesis agent (to determine action and outlook). An orchestrator agent connected all these agents together.

### The Output and Its Incoherence Problem [03:09](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=189s)
The system generated information packets that seemed good at a high level but lacked coherence when examined closely. For example, it correctly identified that prescriptions dropped 18% in a territory and correctly attributed the cause to a payer moving the drug to a lower tier (making it expensive for patients). However, the recommended action—send more sales reps to talk to doctors—didn't address the actual payer coverage problem. Because the action was wrong, the outlook was also wrong. Each agent derived the right fact individually, but no single agent owned the end-to-end picture.

### Why the Multi-Agent Approach Failed [04:42](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=282s)
Subbiah identifies three key issues. First, a language model was being used to determine signals (like sales drops) that should be handled by simple statistical methods. Second, context was lost during handoffs between agents—the driver attribution agent found the right cause, but the synthesis agent didn't understand the weightage of insurance coverage going down. Third, there was no shared understanding of business domain knowledge; agents didn't understand metrics like TRx (number of prescriptions a doctor writes) or the relationships between them.

### Going Back to the Drawing Board with Claude Code [06:16](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=376s)
Abhilash takes over, explaining their first instinct was to redesign the topology, skills, tools, and handoff schemas. Instead, they stepped back and opened a plain directory with Claude Code, giving it bash and database access plus an actual signal, then observed what it did. This observation session revealed solutions to all three problems they had identified.

### Separating Deterministic Signal Detection [07:17](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=437s)
They observed that the agent was inconsistently deciding on signals—sometimes applying statistical methods, sometimes barely looking at the data, sometimes identifying noise as a signal. They separated this into a pure deterministic workflow with statistical methods, guardrails, thresholds, and prioritization. An automated pipeline scans data, identifies signals for each KPI, detects anomalies and trends, and places signals on a queue. The agent's job becomes investigation, not identification—it wakes up when a signal arrives in the queue.

### Consolidating into a Single Agent [08:20](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=500s)
Observing Claude Code repeatedly writing functions and querying databases, they consolidated the entire process into a single agent that owns reasoning end-to-end. They maintained parallelism but removed distributed reasoning. Sub-agents are still spawned dynamically for focused investigative tasks (e.g., understanding rep activity in a particular region), but the main agent retains control over judgment and synthesis—only the investigation is delegated.

### Building the Knowledge Graph as a Control Plane [09:22](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=562s)
The consolidated agent still lacked business context—it didn't understand entities, domain KPIs, or how they relate. Looking at tables and trying to infer relationships wasn't scalable and often produced non-existent relationships. The team sat with domain experts who had years of pharma commercial analytics experience and built a knowledge graph mapping geographic entities, payers, accounts, brands, KPIs, and their relationships—including how primary, secondary, and tertiary KPIs drive each other.

### How the Agent Navigates the Knowledge Graph [10:55](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=655s)
The knowledge graph is not just a data lookup; it serves as a control plane that dictates what the agent can investigate, what paths it can take, and what hypotheses it can evaluate. For source localization (the "where"), the graph guides the agent through permutations of regions, territories, payers, and accounts. For root cause analysis (the "why"), the graph's KPI relationships help the agent determine which KPIs are driving the problem. Every edge in the graph represents a hypothesis the agent can evaluate, keeping the investigation bounded.

### The Investigation Loop [12:27](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=747s)
The agent follows a loop pattern: it starts with an entity, goes to the graph, looks at the neighborhood, identifies edges as hypotheses, returns to the original data to evaluate each hypothesis against actual numbers, reasons over the findings, and either finds supporting or contradicting evidence. If evidence supports a hypothesis, it traverses further through the graph. This repeats until all hypotheses are exhausted or the root cause is identified.

### Results and Key Takeaways [13:30](https://www.youtube.com/watch?v=u6jJcIFDLE4&t=810s)
After 50+ turns and significant token usage, the agent produces in 20–30 minutes what an analyst previously produced in 3–4 weeks. The four key takeaways: (1) Don't introduce human or design constraints into architecture—let it be derived. (2) Complex workflows have both deterministic and agentic parts—don't let agents run deterministic parts. (3) Have one agent own reasoning end-to-end, even if it delegates sub-tasks. (4) Treat the knowledge graph as a control plane for navigation and decision-making, not just a lookup layer.
