## TL;DR
Brian Douglas shares his journey of training AI agents on custom code by capturing, analyzing, and learning from agent session data. Using a Pokémon speedrunning experiment as a case study, he demonstrates how to turn "dead tokens" into reusable skills and explores fine-tuning small language models on personal coding session data.

## Key Takeaways
- **Capture your agent sessions**: Tools like Claude Code and Codex store session data locally for only 30 days, meaning valuable learning data is constantly being lost.
- **Turn dead tokens into skills**: By recording agent sessions in a database (via the open-source tool Tapes.dev), you can analyze past interactions and generate reusable "skills" from successful patterns.
- **Self-healing agent loops**: By feeding observations back to agents, you can create self-improving loops where agents learn from their own mistakes—demonstrated through a Pokémon Red speedrunning experiment.
- **Cheaper models can work at scale**: For repetitive, well-defined tasks (like linting or documentation generation), you can use cheaper models like Haiku instead of expensive flagship models.
- **Specialized Fine-Tuning (SFT) is accessible**: You can fine-tune a small language model (like a 4B parameter model) on your own session data using tools like QLoRA and Unsloth on a local gaming rig.
- **Direct Preference Optimization (DPO) is expensive and difficult**: DPO requires significant VRAM (32GB+) and yields middling results on small models; it's not recommended unless you have a dedicated research use case.
- **Data ownership matters**: Companies are selling your session data to third parties (e.g., Cursor's deal with SpaceX). You should collect and derive value from your own data.
- **Knowledge transfer enables model choice**: Once you've captured your learnings as skills, you gain the freedom to switch between different AI models and harnesses without losing accumulated context.

### The Pokémon Speedrunning Case Study [01:34](https://www.youtube.com/watch?v=o-IunU6b1t8&t=94s)
Brian introduces a case study where he used AI to speedrun Pokémon Red on a Game Boy emulator. He used PyGameBoy, a Python library for headless Game Boy emulation, combined with Claude Code (referred to as Quad Code/Codex) as a general-purpose harness. His goal was to run an agent for 1,000 turns, capturing screenshots every 10 turns to identify learning patterns. He faced initial challenges, such as the agent not knowing it needed to talk to NPCs (like the player's mom) for context clues. By enforcing a rule that the agent couldn't use the internet to learn the game, he forced a self-learning loop.

### Observational Memory and Self-Healing Loops [06:11](https://www.youtube.com/watch?v=o-IunU6b1t8&t=371s)
To solve the Pokémon challenges, Brian dumped all raw context into a Tapes database (initially SQLite, later migrated to Postgres for concurrency). He introduced the concept of "observational memory"—having the agent write human-readable markdown observations at the end of sessions. He also used an "observer state JSON" to track specific game mechanics, like the 7-second cooldown when passing through doors. Through Socratic questioning and anomaly detection (running 10 simultaneous agents via Kafka), the agent eventually learned to speedrun to Professor Oak's cabin in three seconds.

### The Problem with Dead Tokens [09:18](https://www.youtube.com/watch?v=o-IunU6b1t8&t=558s)
Brian draws a parallel between the hidden nuances in legacy codebases and the complexity of Pokémon. He points out that developers are currently "shooting from the hip" with AI, getting new token windows every 5 hours but never learning from the previous sessions. Claude Code and Codex store sessions locally for only 30 days before deletion. He emphasizes that we are paying for tokens but getting no long-term output or value from the session data.

### Sweeper Agent and Data Ownership [11:21](https://www.youtube.com/watch?v=o-IunU6b1t8&t=681s)
Brian introduces "Sweeper Agent," a system that runs 10 parallel agents in separate VMs (using Stereos) to clean up old code, fix lint errors, and write documentation. He shares a story about getting banned from Claude for running 10 parallel sweeper agents, which was resolved after he emailed them a link to his blog post. He uses this to highlight a broader point: companies are selling your session data (mentioning Cursor's $10M deal with SpaceX), so you should collect and utilize it yourself.

### Tapes.dev and Skill Generation [16:30](https://www.youtube.com/watch?v=o-IunU6b1t8&t=990s)
Brian explains the Tapes skills pipeline. Tapes.dev captures agent sessions and stores them in a Merkle DAG structure (similar to Git commits). He mentions a tool called "Git Blamer" he built on LanceDB, but pivots to the "check the tapes" skill, which allows him to query past sessions for context—like reconstructing prompts to share with a designer. Tapes Deck provides a visualization layer for these sessions, showing tool calls and skill invocations. Because the data is vectorized, you can use natural language search to find relevant sessions and generate new skills, even using older, cheaper models like GPT-4o.

### Specialized Fine-Tuning (SFT) vs. DPO [22:11](https://www.youtube.com/watch?v=o-IunU6b1t8&t=1331s)
Brian transitions to fine-tuning, referencing his work at Continue on the "Next Edit" model. He explains two approaches:
1.  **Specialized Fine-Tuning (SFT)**: Compared to writing notes in the margins of a book. He fine-tuned a Qwen 4B model using QLoRA and Unsloth on his session data. It worked well for bespoke, specialized tasks (like onboarding to a specific project) but is not a daily driver replacement.
2.  **Direct Preference Optimization (DPO)**: Compared to CliffNotes—it always picks the best choice. This is expensive and requires significant VRAM. He had to upgrade to an RTX 5090 (and later borrowed H100s) to run it. The results on a 4B model were "not good" and "not even worth it." He does not recommend DPO unless you are a researcher with a specific, expensive problem.

### Conclusion and Next Steps [28:53](https://www.youtube.com/watch?v=o-IunU6b1t8&t=1733s)
Brian summarizes the workflow: Step one is to capture sessions (pointing to doc.cloud/sessions and tapes.dev). Step two is knowledge transfer—moving from solo player to multiplayer by sharing skills between machines and harnesses. This unlocks "model choice," allowing you to switch from Opus to Haiku or other models as needed, ultimately leading to cost optimization. He concludes by reminding attendees to trace data and reinforce learnings.
