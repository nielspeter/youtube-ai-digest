## TL;DR
Simon Martinelli shares lessons from adopting spec-driven development with AI for enterprise business applications, advocating for SysML use cases and entity/domain models as specifications that both stakeholders and AI can understand. He introduces his "AI Unified Process," argues that architecture choices like self-contained systems profoundly affect AI-assisted development, and demonstrates how well-crafted skills, guardrails, and MCP servers can yield near-deterministic code generation.

## Key Takeaways
- **Specs reduce nondeterminism** in AI-generated code, but specs alone aren't enough—you need a full harness of skills, guidelines, MCP servers, and architecture documentation.
- **SysML use cases** (created by Ivar Jacobson in 1987) combined with an entity/domain model serve as a shared language between business stakeholders and AI, and AI already understands them well due to their long history.
- **Use cases are richer than user stories**: a single use case typically encompasses multiple user stories, with preconditions, post-conditions, a main success scenario, and alternative flows.
- **The AI Unified Process** covers the full software development lifecycle for enterprises, unlike more developer-centric tools such as Amazon Kiro, GitHub Spec Kit, or Behave Method.
- **Architecture matters enormously**: naive microservices create a "distributed big ball of mud" where gathering context for AI is extremely difficult; monoliths are too large; **self-contained systems** (vertical slices with UI, logic, and database together) provide the ideal context boundary for AI.
- **Software modernization** should reverse-engineer use cases and entity models from existing code and documentation, then generate new code—enabling new features to be added during modernization rather than pure lift-and-shift.
- **Skills must match the stack and outcome**: different customers use different frameworks (React, Vaadin, Angular, Quarkus, Spring Boot), so skills are customized per project and shared across the organization.
- **Never let AI create a project from scratch**—use scaffolding tools like start.spring.io to get current dependencies and conventions; letting AI bootstrap projects wastes tokens and risks outdated results.
- **Team structure shifts**: team sizes shrink from 5–7 developers to 1–2 per self-contained system, sprints are replaced by continuous Kanban-style flow, and pull requests give way to trunk-based development with peer review.
- **Work shifts left** toward requirements engineering and specification—product owners and requirements engineers now spend more time than developers, who primarily automate and oversee generation pipelines.

## Detailed Breakdown

### From Volunteer Management to Spec-Driven Development [00:33](https://www.youtube.com/watch?v=odbNXv9xXjc&t=33s)
Martinelli opens with a personal story about building a volunteer management system for his sports club in Switzerland using AI tools (Windsurf) in fall 2024. When a music festival asked for the same system, he realized he had no understanding of what he'd implemented or how to adapt it. This pain point led him to discover spec-driven development through an article by Simon Maple on AI Native Dev.io, prompting him to develop an approach tailored to enterprise business applications rather than tools or products.

### The AI Unified Process [03:12](https://www.youtube.com/watch?v=odbNXv9xXjc&t=192s)
Martinelli situates his approach within the broader spec-driven development ecosystem, distinguishing his process-centric "AI Unified Process" from tool-centric approaches like Amazon Kiro, GitHub Spec Kit, Behave Method, and Tessl's spectrum tools. He argues existing tools are too developer-centric and don't cover the full software development lifecycle needed in large enterprises with multiple roles. His process starts with a vision, moves through requirements gathering (referencing the International Requirements Engineering Board's new AI micro-credential), and centers on two key specification artifacts.

### SysML Use Cases and Entity Models [04:17](https://www.youtube.com/watch?v=odbNXv9xXjc&t=257s)
The core of Martinelli's approach uses **SysML use cases**—originated by Ivar Jacobson in 1987 and later absorbed into UML—as the behavioral specification, paired with an entity/domain model for data structure. He chose these because they historically served as effective communication tools between stakeholders and developers, and AI understands them well due to decades of training data. Unlike tool-based approaches that generate plans and tasks from PRDs, Martinelli skips the plan/task phase and generates code directly from use cases and entity models. Additional information like UI designs (via Figma MCP servers) or API specs supplement the use cases.

### Test-Driven Development vs. UI-First [06:53](https://www.youtube.com/watch?v=odbNXv9xXjc&t=413s)
Martinelli addresses the ordering of code and test generation. For APIs, he advocates test-driven development—starting with tests that then drive code generation. For full-stack work with UIs, TDD is harder because the UI appearance must be settled before meaningful tests can be written. He also introduces **risk-based review**: different modules in an ERP system have different criticality (e.g., inventory failures are tolerable; order management failures cost money), so review depth should vary accordingly.

### Greenfield Projects and Skills [08:56](https://www.youtube.com/watch?v=odbNXv9xXjc&t=536s)
On greenfield projects, requirements engineers, product owners, or business analysts create the entity model and use cases, potentially deriving them from requirements. A definition-of-done phase follows where engineers and requirements staff jointly decide when specs are sufficient. Then an agent—equipped with skills, MCP servers, guidelines, and guardrails—generates code. Martinelli stresses that skills must match the target stack and outcome; he currently works with six customers, each using different framework combinations (React + Spring Boot, Vaadin + Spring Boot, Angular + Quarkus), requiring distinct skill sets.

### Software Modernization via Reverse Engineering [10:29](https://www.youtube.com/watch?v=odbNXv9xXjc&t=629s)
Most of Martinelli's work is modernization, not greenfield. His process extracts use cases and entity models from existing code, tests, and scattered documentation (Confluence, Jira, etc.). Business people review these specs, and new code is then generated. He explicitly rejects direct COBOL-to-Java transformation ("lift and shift"), noting it failed 30 years ago and fails now. Instead, going through specs allows new features to be integrated during modernization—something previously forbidden to avoid introducing bugs. End users respond positively because the system genuinely improves, not just changes technology.

### Why Use Cases Over User Stories [12:04](https://www.youtube.com/watch?v=odbNXv9xXjc&t=724s)
Martinelli argues use cases are superior to user stories because they are well-defined and AI already knows how to write them. A use case contains preconditions, post-conditions, a main success scenario, and alternative flows—making it essentially a collection of related user stories. Their larger scope provides more context and structure than isolated user stories.

### Spring Pet Clinic Demo and Reverse Engineering [13:07](https://www.youtube.com/watch?v=odbNXv9xXjc&t=787s)
Martinelli demonstrates using the classic Spring Pet Clinic sample application. He shows a reverse-engineered UML use case diagram with two actors (visitor and clinic user) and grouped use cases across modules (welcome page, owner management, pet management, visit management). He also shows the entity model derived from the database schema. From these specs, he invokes an "implement" skill in Claude Code, which generates the doctor-listing feature in about 90 seconds. He emphasizes that in his projects, developers don't prompt—they use and iterate on skills.

### Architecture's Impact on AI-Assisted Development [17:47](https://www.youtube.com/watch?v=odbNXv9xXjc&t=1067s)
Martinelli argues architecture profoundly affects how well AI works. Naive microservices create a "distributed big ball of mud"—one insurance customer has ~500 microservices and ~500 micro front-ends in an N-to-M relationship, making it nearly impossible to gather sufficient context for AI. Pure monoliths are too large for AI context. He advocates **self-contained systems**: vertical slices containing UI, business logic, and database in one repo/project, allowing AI to work with full context. Different verticals can use different technologies (e.g., Vaadin for inventory, React for order management).

### Guardrails and Near-Deterministic Output [23:06](https://www.youtube.com/watch?v=odbNXv9xXjc&t=1386s)
Martinelli shares key guardrails: never let AI create projects from scratch (use scaffolding tools like start.spring.io instead); avoid bloated system prompts (citing an ETH Zurich study linking larger prompts to more hallucinations); use arc42 architecture documentation; invest in skills; and use MCP servers with vector search for large in-house framework documentation. With good iteration on these elements, results become near-deterministic—deleting and regenerating yields essentially the same output. Human review depth should scale with risk.

### Team and Process Transformation [20:56](https://www.youtube.com/watch?v=odbNXv9xXjc&t=1256s)
The approach transforms team structure: from 5–7 developers down to 1–2 per self-contained system (preferably two for knowledge exchange). Sprints are eliminated in favor of continuous Kanban-style flow using use cases to track progress. Pull requests are replaced by trunk-based development with ongoing peer review—two developers work together and explain their AI-generated code to each other.

### Conclusion: Specs, Sustainability, and Shifting Left [25:45](https://www.youtube.com/watch?v=odbNXv9xXjc&t=1545s)
Martinelli concludes that specs reduce nondeterminism but need a full harness to work. He sees specs as potentially sustainable—reverse-engineered specs could regenerate applications in different technologies or UI paradigms (even chat interfaces), and business people could modify system behavior without developers. However, work shifts left: requirements engineering and specification now take the most time. In a Swiss parliament project, two product owners/requirements engineers work on specs while Martinelli alone handles code automation, building pipelines so markdown spec changes trigger automatic regeneration.
