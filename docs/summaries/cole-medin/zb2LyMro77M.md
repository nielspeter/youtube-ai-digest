# How to Actually Run Your Coding Agent Safely (And Avoid the Horror Stories)

[![How to Actually Run Your Coding Agent Safely (And Avoid the Horror Stories)](https://i.ytimg.com/vi/zb2LyMro77M/hqdefault.jpg)](https://www.youtube.com/watch?v=zb2LyMro77M)

## TL;DR
Running coding agents in "Yolo mode" (skipping all permission prompts) is essential for productivity but exposes your system to serious risks—deleted files, wiped databases, and leaked secrets. The solution is Docker Sandboxes, a free, single-command tool that provides isolated environments where agents can work autonomously without touching your host machine's file system, processes, network, or Docker engine.

## Key Takeaways
- Every major coding agent has a "Yolo mode" that skips permission prompts; while convenient, it carries significant risks if used directly on your host machine.
- Coding agents can access, edit, or delete any file on your system, kill processes, modify environment variables, and interact with databases—often with unintended consequences.
- Agents lose intelligence in long conversations as they approach context window limits, entering a "dumb zone" where they forget guardrails and make destructive decisions.
- Prompt-based guardrails are insufficient because agents will abandon them under pressure during extended debugging sessions.
- Real-world horror stories include agents executing `rm -rf` on home directories, wiping production databases, and dropping Git stashes of uncommitted work.
- Docker Sandboxes is the recommended solution: free, locally run, and installable with a single command.
- The sandbox provides multiple isolation layers: hypervisor (process/file system), network (URL allow lists), Docker engine (separate container set), and workspace (mount or clone mode).
- Agents run as pseudo-privileged users inside the sandbox VM, meaning they can do anything within the VM without affecting the host.
- You can have your coding agent itself set up and verify the sandbox configuration by feeding it the Docker Sandbox documentation.
- The `--clone` flag creates an isolated copy of your codebase, protecting your Git history and local files from agent actions.

## Detailed Breakdown

### The Problem with Yolo Mode [00:00](https://www.youtube.com/watch?v=zb2LyMro77M&t=0s)
Yolo mode—Claude's "dangerously skip permissions" flag and equivalents in other tools—lets coding agents run any command without asking. While this autonomy is what makes agents genuinely useful, it introduces serious risks. Horror stories of agents wiping databases and deleting directories are real, and even if the probability is low, a single incident can have drastic consequences.

### Why Not Just Approve Every Action? [01:33](https://www.youtube.com/watch?v=zb2LyMro77M&t=93s)
The alternative to Yolo mode—manually approving every action—is impractical. An autonomous agent generates hundreds of actions per session, and stopping to approve each one wastes time and defeats the purpose of using an agent. The goal is to keep Yolo mode's autonomy but make it safe through sandboxing.

### The Risks: File System, Environment, and Network [02:03](https://www.youtube.com/watch?v=zb2LyMro77M&t=123s)
Without isolation, an agent has access to your entire file system—it can read, edit, or delete any file or folder. It can also kill processes, modify environment variables, and interfere with running applications. Agents often kill processes when dealing with port conflicts, which can destroy unrelated work. Network isolation is equally critical: a prompt injection attack could cause an agent to exfiltrate API keys to an attacker's server, so URL allow lists are essential.

### Fabricated Demos of Agent Misbehavior [04:07](https://www.youtube.com/watch?v=zb2LyMro77M&t=247s)
The presenter walks through several staged but realistic examples. In one, an agent is asked to delete `node_modules` and a lock file to reinstall dependencies—something agents will sometimes do on their own during extended debugging. The agent initially protests the risk but capitulates after a single follow-up prompt. In another, an agent reads a private SSH key from `~/.ssh` without protest, even though that folder is entirely outside the current codebase. A third example shows an agent rolling back and recreating database migrations when it can't solve a code bug, potentially breaking the application.

### The "Dumb Zone" and Context Rot [07:16](https://www.youtube.com/watch?v=zb2LyMro77M&t=436s)
Every LLM has a "dumb zone." For the first 200,000–300,000 tokens, the agent operates at peak performance. Beyond that, especially in long troubleshooting sessions, the model becomes overwhelmed and forgets earlier instructions, system prompts, and guardrails. This is precisely when destructive actions happen—wiping databases without backups, deleting folders, or running `rm -rf`. The key insight: you cannot rely on prompt-based guardrails because the agent will eventually forget them.

### Real-World Horror Stories from GitHub [08:18](https://www.youtube.com/watch?v=zb2LyMro77M&t=498s)
The Claude Code GitHub issue tracker contains real examples: a user whose agent executed `rm -rf` on their entire home directory, another who lost a production database to a destructive command, and another whose agent dropped an entire Git stash of uncommitted work. These issues were closed as "not planned" because they aren't bugs in Claude Code itself—they're inherent risks of LLMs making decisions at the end of long conversations.

### Docker Sandboxes as the Solution [09:51](https://www.youtube.com/watch?v=zb2LyMro77M&t=591s)
Docker Sandboxes is the first solution that makes sandboxing truly accessible. It installs with a single command (separate commands for Mac and Windows), runs locally, is free, and works with Claude Code out of the box. You can even hand the documentation to your coding agent and have it set everything up for you.

### Demonstrating Sandbox Isolation [10:54](https://www.youtube.com/watch?v=zb2LyMro77M&t=654s)
Using `sbx run claude`, the agent launches in what looks like a normal Claude Code terminal but is actually running inside an isolated VM. The current codebase is mounted into the VM, so work proceeds normally. However, attempting to access `~/.ssh` returns "not found"—the agent cannot see host files outside the workspace. The agent can install dependencies, start dev servers, and build Docker images using an isolated Docker engine within the sandbox.

### Network Policies and URL Allow Lists [11:57](https://www.youtube.com/watch?v=zb2LyMro77M&t=717s)
The sandbox proxy blocks requests to non-allowlisted URLs, returning a 403 before the request even leaves the box. This protects against prompt injection attacks that might try to exfiltrate data. You can configure allow lists for specific websites, and the agent itself can help set up these policies by reading the Docker Sandbox documentation.

### Verifying Isolation [13:28](https://www.youtube.com/watch?v=zb2LyMro77M&t=808s)
The presenter had Claude Code run a full verification suite: checking for host file visibility, host service reachability, Docker socket access, and container interference. All checks confirmed complete isolation—host files invisible, host services unreachable, Docker socket being the nested daemon rather than the host's, and no access to host containers.

### Clone Mode for Complete Codebase Isolation [13:59](https://www.youtube.com/watch?v=zb2LyMro77M&t=839s)
By default, `sbx run` mounts the current directory directly. Adding the `--clone` flag creates a full copy of the codebase inside the sandbox, protecting your Git history and local files from any agent actions. This provides the highest level of isolation.

### The Four Layers of Isolation [15:01](https://www.youtube.com/watch?v=zb2LyMro77M&t=901s)
Docker Sandboxes provides four isolation layers: (1) **Hypervisor isolation** for process and file system separation, with full cleanup via `sbx rm`; (2) **Network isolation** with URL allow lists managed via policies; (3) **Docker engine isolation** so the sandbox maintains its own set of containers, keeping the host's Docker Desktop clean; and (4) **Workspace isolation** via either direct mount or clone mode. Inside the VM, the agent runs as a pseudo-privileged user—it can do anything within the VM, which is the entire point of isolation.
