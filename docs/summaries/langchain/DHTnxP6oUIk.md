# Build a secure computer for your agent

## TL;DR
As AI agents increasingly write and execute code rather than merely answering questions, they require dedicated, secure computing environments to run safely. LangChain's LangSmith Sandboxes provide fast, isolated microVMs that allow agents to execute untrusted code, manage state, and scale to thousands of instances without risking local or production infrastructure.

## Key Takeaways
- Agents have evolved from answering questions to writing and executing real code, making a secure "computer" a necessity for production-level agent workflows.
- Running agent-generated code locally is risky; incidents like accidental file deletions and supply chain worms highlight the dangers of arbitrary code execution.
- LangSmith Sandboxes are isolated, disposable microVMs with kernel-level isolation, meaning they do not share a kernel with the host like standard Docker containers do.
- Sandboxes spin up in under a second and can scale from zero to thousands of parallel instances to handle dynamic, user-facing workloads.
- An "O proxy" acts as a man-in-the-middle to control all network egress, keeping credentials off the runtime and preventing data exfiltration to malicious domains.
- Sandboxes support long-running, interruptible tasks with persistent state, allowing users to pause, resume, snapshot, and fork environments.
- The product integrates natively with LangChain agents and open-source frameworks like OpenSWE, with support for bringing custom Docker images and external mounts (S3, GCS, GitHub).
- LangSmith offers robust access controls (ABAC/RBAC) and is actively developing self-hosted options for highly regulated enterprises.
- Free sandbox credits (5 LCUs / 1 LSU, equating to ~100 minutes) will be available starting the Friday following the webinar for all plans, including the free tier.

### The Rise of Coding Agents and Use Cases [03:19](https://www.youtube.com/watch?v=DHTnxP6oUIk&t=199s)

### Security Risks of Local Execution [08:03](https://www.youtube.com/watch?v=DHTnxP6oUIk&t=483s)
Running agent-generated code locally is highly risky. The speaker cites recent incidents, including a Codex update deleting files on a local machine, a self-propagating supply chain worm from npm stealing developer tokens, and remote execution attacks on platforms like Naden and Google's Project IDX IDE. Handing an agent arbitrary code execution rights on local infrastructure is a major security liability, prompting the need for a better solution.

### Defining the Sandbox [09:05](https://www.youtube.com/watch?v=DHTnxP6oUIk&t=545s)
A sandbox is defined as an isolated, disposable environment for agent code that is separate from core infrastructure. It provides its own file system and memory, with granular network controls to block malicious websites. Sandboxes must be highly parallelizable and disposable—designed for one or two tasks before being thrown away. Unlike a Mac Mini, they provide a full environment with a shell and package installation capabilities without the overhead of managing physical hardware.

### LangSmith Sandboxes Features [10:37](https://www.youtube.com/watch?v=DHTnxP6oUIk&t=637s)
LangSmith Sandboxes address the challenges of running agents at scale. They feature a median spin-up time of under a second and can scale to thousands of instances dynamically. For security, they use microVMs with kernel-level isolation to prevent container escapes. A unique "O proxy" manages egress, ensuring credentials never touch the runtime. Sandboxes also support long-running tasks with persistent state, idle TTLs for automatic spin-down, and snapshot/fork capabilities to recover from agent mistakes.

For cloud architecture, LangSmith Sandboxes are recommended over Kubernetes due to Kubernetes' scheduling latency (around 8 seconds), which is too slow for user-facing applications. ### Q&A: Enterprise, Docker, and Security Controls [28:02](https://www.youtube.com/watch?v=DHTnxP6oUIk&t=1682s)
For highly regulated enterprises, a self-hosted, air-gapped version of the platform is in development. The speakers argue against using standard Docker, as containers share a kernel with the host and require dropping privileges, which limits agent capabilities. For detonating malware or clicking malicious links, the O proxy's network-level controls are the most powerful defense, preventing data exfiltration to non-allowlisted domains. Users can bring their own Docker images to ensure reproducible and portable runtimes.

### Q&A: State Persistence and Competitors [31:41](https://www.youtube.com/watch?v=DHTnxP6oUIk&t=1901s)
Sandbox file system state persists across sessions, and users can configure auto-idle policies. Memory snapshotting saves running processes, and users can fork or clone sandboxes from snapshots. Compared to competitors like Daytona, LangSmith differentiates itself through the O proxy and deep integrations with the broader LangSmith ecosystem, such as Context Hub.
