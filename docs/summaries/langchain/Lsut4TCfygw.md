# LLM Wikis and how to give your agents memory

## TL;DR
A panel from LangChain, Chroma, and Dosu explores "LLM wikis" — agent-generated knowledge bases designed to make AI more efficient by caching compiled knowledge. While participants agree wikis should be human-interpretable but optimized for agents, they debate the ideal structure, maintenance challenges, and whether the "wiki" abstraction itself will evolve into something more like a cache or index.

## Key Takeaways
- **Wikis as caches:** Devin Stein frames LLM wikis as a "cache" of compiled knowledge, arguing you should only store what is frequently accessed or expensive to recompute — not everything.
- **Human-interpretable, agent-optimized:** All panelists agree wikis should remain readable by humans (in natural language, not latent vectors) for steerability, but should be optimized for agent consumption (concise, highly referential, terse).
- **Maintenance is the hardest problem:** Generating an initial wiki is relatively easy; keeping it updated concisely as context grows, identifying *where* to update, and managing cost/ROI at scale remain unsolved challenges.
- **Measuring ROI is difficult:** Token efficiency (fewer tool calls, lower cost) and output consistency are the primary metrics, but designing proper evals is still an open problem.
- **Eventually consistent knowledge:** Devin advocates building self-correcting loops where agents detect and fix incorrect wiki entries over time, rather than requiring human approval of every change.
- **File systems vs. databases:** Jeff Wang argues file systems are inadequate for long-term wiki storage due to poor version control, access control, and discovery; databases will likely re-emerge as the proper substrate.
- **Search should be multi-modal:** Agents should be given a blend of file system tools, full-text search, and semantic/dense vector search, and trusted to choose the right tool for the job.
- **Beyond coding:** While current use cases are engineering-focused, wikis could become the "state layer" of an organization, applicable to GTM, customer success, and general knowledge work.

## Detailed Breakdown

**[00:00] Intro & Demos**
Harrison (moderator) introduces the last-minute webinar on LLM wikis and memory, sparked by conversations with Devin and Jeff. He notes the topic's growing relevance, partly fueled by Andrej Karpathy's commentary. Each panelist gives a brief demo/intro:
- **Brace (LangChain):** Demonstrates *Open Wiki*, a CLI-based memory agent that auto-generates and maintains docs from connected sources (Notion, Gmail, Slack) via a daily cron job. [02:07]
- **Devin Stein (Dosu):** Shows how Dosu plugs into coding agent workflows via MCP, capturing learnings as human-readable but agent-optimized "knowledge index" pages, with analytics on usage. [04:13]
- **Jeff Wang (Chroma):** Frames wikis as a "new kind of index" for traversing unstructured data, noting Chroma's interest in the space and the difficulty of building/maintaining wikis. [06:17]

**[08:54] Defining a Wiki**
Harrison offers a naive definition: files in a directory generated by an agent for an agent. Jeff pushes back, saying that's overfit to technology; a wiki is more abstractly "hyperlinked pages" intended for information consumption. Brace and Devin agree, with Devin emphasizing the "compiled knowledge" or "cache" analogy inspired by Karpathy.

**[11:30] The Cache Analogy & Devin's Hot Take**
Devin elaborates: wikis contain "necessary knowledge" (info that exists nowhere else) and "compiled knowledge" (cached for efficiency). His hot take: current wiki implementations are too human-oriented (front-to-back reading structure). He argues for an agent-first approach focused on token efficiency, with aggressive eviction of stale or low-value entries.

**[14:05] For Agents or Humans?**
Brace believes wikis should move toward being primarily for agents, with humans interfacing via agents. Jeff counters that wikis must remain in English (not latent vectors) to preserve human steerability and control — they are for both. Devin distinguishes between "human-interpretable" and "human-audience," arguing for the former while optimizing for the latter.

**[15:47] Human-in-the-Loop**
- **Devin:** Dosu ties knowledge changes to PR review ("knowledge reviews"), but doesn't require approval for every change. Knowledge is "eventually consistent" — agents self-correct errors over time.
- **Brace:** Open Wiki's code mode puts wikis in markdown files requiring PR review; general-purpose mode writes freely but uses "open questions" and planned changelogs for human visibility.
- **Jeff:** We're in the "prompt engineering era" of wikis; human judgment is still crucial for hill-climbing, but eventually agents should learn to interview humans when they need help.

**[21:22] Hardest Parts of Building Wikis**
- **Brace:** Prompting for updates — identifying *where* to update in a large wiki and concisely modifying vs. inserting.
- **Devin:** At scale (tens of millions of lines of code), identifying what's important is hard; maintenance is expensive; measuring ROI/observability is difficult.
- **Jeff:** Harness engineering for accuracy + cost management — a multivariate optimization problem.

**[25:00] Measuring Impact**
Brace suggests running benchmarks with/without docs to measure token efficiency (fewer tool calls, fewer tokens). Devin shares that internal evals show runs with wiki entries cost ~50% less, and outputs are more consistent because the wiki provides a stable "seed." Jeff notes two categories: improving existing workflows (cheaper, faster, fewer interventions) and enabling net-new capabilities (long-horizon tasks, proactive actions, agent collaboration).

**[30:37] Devin's Hot Take Revisited**
Devin clarifies: the wiki *concept* is fine, but the shape should change for agents — more terse, highly referential (agents can dig deeper via links), with aggressive eviction of unused entries. Jeff adds questions about page length, progressive disclosure, active vs. passive linking, and backlink cleanup as open problems.

**[33:15] Q&A: Knowledge Work & Graphs**
- On knowledge workers: Brace sees general-purpose wikis feeding into chat interfaces; Jeff boldly claims the wiki will become the "state, communication, coordination, and execution layer for every company."
- On knowledge graphs: Panelists agree wikis are graph-like (linked pages) but more flexible than rigid RDF/Neo4j triple stores; you want linking, not rigid noun-verb-noun structure.

**[37:22] Q&A: Vector Databases & Search**
Jeff dismisses "vector database" as a made-up term, recentering Chroma's mission as storing massive unstructured data and making it searchable. On exposing wikis to agents: Brace recommends a mix of agentic file system search, full-text, and semantic search. Jeff strongly advocates giving agents all search tools (grep, sparse, dense) and trusting them to choose — "the bitter lesson applied to search." He also argues file systems are inadequate for long-term wiki storage (poor version/access control, bad discovery) and that databases will be rediscovered as the proper substrate.
