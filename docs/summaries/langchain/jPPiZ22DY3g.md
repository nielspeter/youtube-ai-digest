# The Art of Loop Engineering: How to Build Agents That Improve Over Time

[![The Art of Loop Engineering: How to Build Agents That Improve Over Time](https://i.ytimg.com/vi/jPPiZ22DY3g/hqdefault.jpg)](https://www.youtube.com/watch?v=jPPiZ22DY3g)

## TL;DR
Sydney, a product manager at LangChain, presents a framework called "loop engineering" for building AI agents that improve over time. The webinar outlines four nested loops—the core agent loop, verification loop, event-driven loop, and self-improvement (hill climbing) loop—that progressively add automation, reliability, system integration, and autonomous learning to agent systems. The core message is that the true power of agents comes not just from the models themselves, but from the composable loops built around them.

## Key Takeaways
- **Agents are non-deterministic software:** Unlike traditional code, the "truth" of an agent's behavior lives in its execution traces, making observability and trace analysis foundational to improving them.
- **Four nested loops of agent engineering:** The core agent loop (action-taking), verification loop (grading/reliability), event-driven loop (system triggers), and hill climbing loop (self-improvement) are composable and build on each other.
- **Verification loops add reliability:** By using a grader to check agent output against explicit criteria, developers can use cheaper, less powerful models for complex tasks while maintaining quality.
- **Event-driven loops drive adoption:** Agents deliver the most value when triggered directly within the systems where users work (e.g., Slack, GitHub, email), rather than requiring users to switch contexts.
- **Self-improvement requires traces and evals:** The most powerful loop uses a background "helper agent" (LangSmith Engine) to analyze traces, detect failure modes, and autonomously update prompts, tools, memory, and skills.
- **Human-in-the-loop remains essential:** Automation does not mean removing humans; it means applying human judgment only where it adds the most value (e.g., sensitive tool calls, PR reviews, or harness code changes).
- **Optimize memory and prompts before architecture:** When improving an agent, it is generally better to update memory, skills, and prompts first, only changing the runtime architecture if deterministic steps are being missed.
- **Evals are the foundation of confidence:** You cannot confidently merge changes to an agent's core harness without a solid eval suite to check for regressions or improvements.

### Why Loops Matter [03:15](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=195s)
Sydney explains that agents are useful because they take action, but their usefulness depends on the harness built around the model. Building robust loops into this harness ensures continuous integration, allowing agents to improve with every run. The core primitive of the LangChain agent development life cycle is loops.

### Loop 1: The Core Agent Loop [04:49](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=289s)
The first loop is the basic action-taking loop: a model receives context, calls tools, receives observations, and continues until the task is done. Engineering this loop involves selecting the right model for the task complexity and optimizing tool descriptions. Sydney illustrates this with LangChain's internal "docs improvement agent," which clones a repo, reads/writes docs, and opens a PR.

### Loop 2: The Verification Loop [06:56](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=416s)
Also known as a "slash goal" loop, this loop adds reliability via a grader that scores the agent's output against rubric criteria. If criteria aren't met, the output is fed back into the agent loop. This is particularly useful for getting good results from cheaper models. LangChain's open-source coding agent, Decode (built on Deep Agents), offers unique goal/rubric middleware that generates transparent criteria for review. For the docs agent, the grader ensures all links resolve and CI passes before notifying a human reviewer.

### Loop 3: The Event-Driven Loop [10:05](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=605s)
Sydney argues this loop is what makes agents personally useful because it embeds them in the systems where conversations happen (e.g., Slack, GitHub, email). Triggers can be scheduled (like a daily calendar summary) or event-driven (like receiving an email). The loop completes when the agent updates the system (e.g., merging live docs), which in turn can trigger future events. The docs agent example is expanded: a Slack message triggers the agent, which drafts a PR, runs the verification loop, and sends a Slack notification back to the user.

### Loop 4: The Hill Climbing (Self-Improvement) Loop [13:42](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=822s)
Called the most exciting loop, this automates agent improvement by finding signal in traces. After an agent completes a run, a background helper agent (LangSmith Engine) analyzes the trace to detect failure modes (e.g., improper tool arguments, missed context, ignored preferences). Engine can then autonomously update prompts, tools, skills, and memory, porting changes back to the source code. This loop improves procedural elements (like tool descriptions) and semantic elements (like user-specific memory). Sydney stresses that closing this loop requires good evals to ensure changes don't cause regressions.

### Human-in-the-Loop Across All Loops [19:29](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=1169s)
While loops automate work, add reliability, and drive self-improvement, human approval remains critical for sensitive actions. Sydney breaks down where humans fit: approving sensitive tool calls (e.g., booking travel), scoring ambiguous grader criteria, reviewing workflow artifacts (e.g., docs PRs), and approving changes to the agent's core source code.

### Q&A: Memory Systems [23:46](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=1426s)
When asked how to decide what information to save for multiple agents, Sydney points to the Deep Agents memory documentation. She notes that memory is specific to the agent and recommends exploring Open Wiki (built on Deep Agents) for building repository wikis and personal wikis to help structure memory systems.

### Q&A: Graders, Ambiguity, and Latency [26:22](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=1582s)
For ambiguous reward signals, Sydney suggests using human annotation to train the criteria generator. Addressing latency, she notes that loops inherently add latency; developers must trade off between correctness, cost, and model power. Guardrails like timeouts and max cycles can help manage latency.

### Q&A: Human Approval Bottlenecks and Scaling [32:35](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=1955s)
To balance human approval with scaling, Sydney suggests deferring approval to the end-user (e.g., the traveler approves their own flight) and using model discretion to request human review only for sensitive cases. Leveraging integrations like Slack to ping humans ensures approvals happen quickly.

### Q&A: Context Rot, Evals, and Architecture Changes [35:44](https://www.youtube.com/watch?v=jPPiZ22DY3g&t=2144s)
Sydney addresses context rot by recommending prompt compression for older iterations in long-running loops. For building evals without ground truth, she points to a new LangChain blog on "eval engineering" that can autogenerate evals from live traces. Finally, she advises that when improving agents, developers should first try updating memory, skills, and prompts before changing the agent's runtime architecture.
