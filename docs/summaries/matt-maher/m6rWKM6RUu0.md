# Fable vs GPT-5.6 Wasn’t Even Close

[![Fable vs GPT-5.6 Wasn’t Even Close](https://i.ytimg.com/vi/m6rWKM6RUu0/hqdefault.jpg)](https://www.youtube.com/watch?v=m6rWKM6RUu0)

## TL;DR
Matt Maher compares two AI coding models — Fable and GPT-5.6 Soul — by having each build a multiplayer card/board game from a Claude-generated design document. While both produced impressive first builds in roughly 90 minutes, Fable required only 6–7 follow-up messages and about 3.5 hours of total work, whereas Soul needed 65+ messages and roughly 14 hours of active work time to reach a comparable level of polish, making Fable the clear winner for user-facing application projects.

## Key Takeaways
- Both Fable and GPT-5.6 Soul were given nearly identical workflows: a Claude design file, a build-request instruction file, and a "build until complete" prompt in Codex.
- Fable built a Spades game; Soul built a Parcheesi-style game (renamed "Dash"), which was inherently more complex but not dramatically so.
- Both initial builds took about 90 minutes and both produced surprisingly good first versions.
- After the initial build, Matt used a feedback loop of screenshots and voice notes for both projects, feeding them back to the respective agents for a second round of fixes.
- Fable needed only about 6–7 total messages and ~3.5 hours to reach a polished, near-complete state.
- Soul required 65+ messages and ~14 hours of active work time (spread over several days) to reach a comparable level of quality.
- Soul invented game rules that weren't in the spec and had UX issues like overlapping UI elements blocking gameplay.
- Soul solved individual problems well but failed to "tack together" multiple fixes into a cohesive reinvestigation, requiring constant human guidance.
- Matt concludes that for user-facing applications, Fable is far more autonomous and requires dramatically less hand-holding than Soul.
- The final fidelity of both products was close, but the human investment required made them "not even close" in practice.

## Detailed Breakdown

### Context and Motivation [00:00](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=0s)
Matt references a previous video where Fable built a complete Spades game from a design document to a polished deliverable. This video investigates whether GPT-5.6 Soul can do the same thing with a similar but different application, and whether the two models are interchangeable or whether the choice between them truly matters.

### A Fairness Caveat: Spades vs. Parcheesi [01:02](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=62s)
Matt clarifies that the Soul model was given a harder task. Parcheesi (the game Soul was asked to build) has more moving parts than Spades — multiple players, dice rolling, pawn movement across a board, power-up cards, etc. It's not twice or ten times as complex, but it is noticeably more involved, and the results should be interpreted with that asymmetry in mind.

### The Design and Build Setup [02:02](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=122s)
Both projects started with a design generated via claude.ai/design. The design was downloaded as a zip file and placed into a project folder. Matt created a `docs` folder and a `design` folder, dropped the zip inside, and added a separate `request` file specifying parameters like multi-user peer-to-peer functionality, mobile-first design, and cross-platform compatibility. This request file was essentially the same one used for the Fable/Spades build. He then launched Codex with a single prompt: "Set up a goal. Use the game build request file… and build until complete."

### Demonstrating the Two Builds [04:39](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=279s)
Matt shows both running applications. The Spades game built by Fable works as expected — bidding, playing cards, hosting/joining games with a code. The Parcheesi-style game built by Soul (renamed "Dash" by the design step) also runs: players pick characters, roll dice, move pieces around the board, and interact with power-up cards. Both are functional, though what's shown is closer to the post-revision state than the raw first build.

### Time to First Build [06:13](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=373s)
Both initial builds took approximately 90 minutes. Matt emphasizes that Soul's task was more difficult, so a rougher result would have been anticipated — and the results did reflect that added difficulty, but "over-matched" it in ways that became problematic.

### The Feedback Loop: Screenshots and Voice Notes [06:44](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=404s)
After the first build, Matt played each game, opened a microphone, and narrated everything he didn't like or didn't understand. He took 10–12 screenshots per game, placed them in a folder inside the project, and fed that back to the agent with a high-level goal to fix the issues. Both models received this same second-round treatment.

### Divergence After the Second Round [08:18](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=498s)
After the second round, Fable's Spades game was nearly complete — only a few minor adjustments remained. Soul's Dash game, however, still had significant issues: overlapping UI elements blocked clickable squares during gameplay, and the model invented rules not present in the spec (which it acknowledged and reverted when questioned). Matt describes the UI overlap issue as a "complete failure in understanding how the game itself operates" rather than a technical malfunction.

### The Investment Gap [09:18](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=558s)
Matt tallies the total effort. Fable: roughly 6–7 messages and about 3.5 hours of work time. Soul: 65+ messages and approximately 14 hours of active work time, spread across several days. This is pure work time, excluding waiting periods when the model was processing or when Matt was away.

### Soul's Strengths and Weaknesses [11:23](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=683s)
Matt notes that Soul worked exceptionally well on individual problems — spending hours testing and verifying specific fixes — but it never "tacked together" multiple fixes into a holistic reinvestigation. Each issue was solved in isolation, and the model needed Matt to continuously guide it. This contrasted sharply with Fable, which was more willing to run off, make decisions, and push forward autonomously.

### Final Verdict [13:27](https://www.youtube.com/watch?v=m6rWKM6RUu0&t=807s)
Matt concludes that while both models can produce high-fidelity results, the human investment required by Soul was disproportionate. He states that for user-facing application projects, he would "hands down, without question" choose Fable right now. He acknowledges GPT may improve in future releases, but at version 4.6, the two models are "not even close" in terms of autonomous productivity.
