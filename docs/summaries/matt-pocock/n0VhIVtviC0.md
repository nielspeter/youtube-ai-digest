# Don't waste time on specs: /prototype instead

[![Don't waste time on specs: /prototype instead](https://i.ytimg.com/vi/n0VhIVtviC0/hqdefault.jpg)](https://www.youtube.com/watch?v=n0VhIVtviC0)

## TL;DR
Matt Pocock argues against over-investing in detailed specs before letting AI generate code. Instead, he advocates for prototyping—building cheap, throwaway artifacts to answer design and logic questions at higher fidelity—because the leap from a working prototype to production code is far smaller than the leap from a spec to production code.

## Key Takeaways
- Spec-driven development with AI often leads to disappointment because specs can't capture how things should actually look and behave in practice.
- Prototyping and spikes are underused practices that have become dramatically cheaper thanks to AI-assisted code generation.
- "Fidelity" describes how concrete an artifact is: low-fidelity discussions resolve basic questions, while high-fidelity prototypes resolve questions about appearance, behavior, and logic.
- The key trigger for prototyping is when "how should it look?" or "how should it behave?" becomes a central question.
- Prototypes produce concrete artifacts you can react to and iterate on, yielding richer feedback than discussion alone.
- The Wayfinder skill splits large chunks of work into planning sessions with different ticket types, including "grilling" (discussion) and "prototype."
- Prototypes can be iterated on through feedback sessions, combining elements from multiple options into a refined version.
- Prototypes aren't limited to UI work—pure logic prototypes (e.g., interactive terminal apps testing state machines) are valuable for complex back-end work.
- Saving prototypes on throwaway branches gives the implementing agent real, referenceable code to work from.
- The gap between a working prototype and production code is much smaller than the gap between a spec and production code.

## Detailed Breakdown

### The Problem with Spec-Driven Development [00:00](https://www.youtube.com/watch?v=n0VhIVtviC0&t=0s)
Pocock opens by describing a frustrating trend: people working with AI to generate code often pour their energy into creating extremely detailed specs upfront, expecting the AI output to perfectly match their plan. In doing so, they forget that they can still write code during the planning process. Prototyping and spikes—long-standing Agile practices—have fallen out of favor, and Pocock wants to reverse this trend.

### Prototypes Are Cheaper Than Ever [01:02](https://www.youtube.com/watch?v=n0VhIVtviC0&t=62s)
A prototype is defined as throwaway code that answers a question. While Pocock has mixed feelings about the phrase "code is cheap," he acknowledges that the cost of producing code has dropped dramatically. This makes generating quick, throwaway prototypes more effective and affordable than ever. He introduces his "prototype skill" and teases its role within his larger Wayfinder skill.

### The Concept of Fidelity [01:34](https://www.youtube.com/watch?v=n0VhIVtviC0&t=94s)
Pocock explains fidelity as a spectrum. Basic questions—like whether a modal should have cancel and confirm buttons—can be resolved through low-fidelity discussion. But questions about how data should display or how something should behave under certain circumstances require higher fidelity. Some questions can only be answered by prototyping, and since prototypes are now cheap to produce, Pocock wants more discussions to happen at higher fidelity.

### The Wayfinder Skill and When to Prototype [03:05](https://www.youtube.com/watch?v=n0VhIVtviC0&t=185s)
Wayfinder is a skill for planning large chunks of work, splitting them into separate planning sessions with their own tickets. Two key ticket types are "grilling" (chatting with the agent to figure out basic scope) and "prototype." The prototype ticket type is triggered when you need to raise fidelity—creating a cheap, rough, concrete artifact to react to. The clear criteria: use a prototype when "how should it look?" or "how should it behave?" is a key question.

### Prototyping in Action: A TLDraw Search Bar [04:07](https://www.youtube.com/watch?v=n0VhIVtviC0&t=247s)
Pocock demonstrates the prototype skill using his TLDraw-based diagramming app. He needed a search bar for navigating old diagrams with a complex data model (diagrams plus snapshots through time). The prototype generated three options (A, B, C), each encoding different design decisions. Option A grouped snapshots by diagram name with a search box placement he liked. Option B used a sidebar filter approach. Option C displayed everything inline without filters. Each option gave him concrete material to react to and evaluate.

### Iterating on the Prototype [05:41](https://www.youtube.com/watch?v=n0VhIVtviC0&t=341s)
The initial prototyping session cost about 100,000 tokens. Pocock compacted the conversation to retain all prototype information while giving feedback. He dictated that he liked the search box from A and the layout from C, then sent this feedback to iterate toward a combined "D" version. This process creates a super-rich asset with encoded design decisions. Pocock notes he typically saves prototypes on throwaway branches so the implementing agent can reference real front-end code.

### Higher Fidelity Costs More but Delivers More [06:42](https://www.youtube.com/watch?v=n0VhIVtviC0&t=402s)
Pocock acknowledges that higher-fidelity prototypes cost more tokens and time. However, resolving the same questions through low-fidelity discussion would yield less useful answers. He also notes that this prototype was integrated into the live page rather than a throwaway route, which he prefers because it gives a more honest representation of how the code will actually work. The resulting D version successfully combined the search box from A with the inline layout from C.

### From Prototype to Production [08:14](https://www.youtube.com/watch?v=n0VhIVtviC0&t=494s)
Once the prototype is complete, Pocock passes it off to an "AFK agent" (away-from-keyboard agent) to integrate everything, delete old prototype code, and ensure compliance with the original spec. The results are strong because the high-fidelity discussion—based on a live, running version—produced invaluable feedback already baked into a referenceable throwaway branch. Pocock contrasts this with the common complaint that "the AI didn't build what I wanted," attributing such failures to insufficiently high-fidelity discussions.

### Prototypes Aren't Just for UI [08:45](https://www.youtube.com/watch?v=n0VhIVtviC0&t=525s)
Pocold stresses that prototyping isn't limited to front-end work. For complex back-end tasks, questions like "does this state model feel right?" arise constantly. The best approach is often to build a pure logic prototype—such as a tiny interactive terminal app that pushes a state machine through hard-to-reason-about cases. He notes that users have praised this feature, and both UI and logic prototype branches have their own reference docs.

### Final Advice and Recommendations [10:21](https://www.youtube.com/watch?v=n0VhIVtviC0&t=621s)
The core message: the leap from discussion and spec to production-ready code is large, but turning a working prototype into production is relatively simple. Pocock recommends "Shape Up" by Ryan Singer as a highly influential, free online book that changed his approach to building applications. He also promotes his skills newsletter for staying current with his constantly evolving skill set.
