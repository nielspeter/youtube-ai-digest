## TL;DR
Nate Herk shares five practical hacks for keeping an AI operating system (AIOS) accurate and scalable, centered on treating your main `CLAUDE.md` file as a router, having the AI audit itself, automating data ingestion, segmenting knowledge, and forcing the agent to backtrack its own mistakes. He also breaks down four context failure modes (poisoning, bloat, confusion, clash) and two context types (expertise vs. situational) to explain why AI agents hallucinate and how to prevent it. A free "OS audit" skill is offered through his community to help viewers automatically detect and fix routing and data issues in their own projects.

## Key Takeaways
- **Four Context Failure Modes:** AI agents give wrong answers due to *poisoning* (false facts in context), *bloat* (too much data to parse), *confusion* (missing or irrelevant facts causing the AI to guess), and *clash* (conflicting data from different sources).
- **Two Context Types:** *Expertise context* is the persistent rulebook (who you are, policies, system prompts) needed for every run, while *situational context* is just-in-time data (like a specific customer ticket) pulled in only when needed to avoid bloat.
- **CLAUDE.md as a Router:** Instead of just using the main `CLAUDE.md` as a system prompt, treat it as a master routing table that tells the agent exactly where to find data, wikis, tools, and project folders.
- **AI Self-Audits:** Regularly have your agent read through your entire OS to check routing integrity, index truth, freshness, and bloat, then suggest fixes.
- **Automate Data Ingestion:** Set up weekly crons (e.g., for meeting or YouTube transcripts) so your AIOS always has fresh data without requiring manual updates.
- **Segment Knowledge:** Split large, distinct data sets (like YouTube transcripts vs. meeting notes) into separate wikis so the agent searches a smaller, more relevant context window, saving time and tokens.
- **Client Project Strategy:** Keep internal client knowledge inside your main AIOS, but build external client deliverables in separate, client-facing repos to maintain boundaries while still giving your agent context.
- **Backtracking:** When an agent fails to find data it should have, force it to retrace its steps, explain why it failed, and then update the routing rules to prevent future errors.
- **Team-Level Syncing:** Scaling an AIOS to a team or department is currently more of a people and habit problem than a technology problem.

### Expertise vs. Situational Context [04:37](https://www.youtube.com/watch?v=Ek1NBfnnTH0&t=277s)
Using his "Four C's" framework (Context, Connections, Capabilities, Cadence), Nate distinguishes between two types of context. *Expertise context* is the persistent rulebook—policies, business goals, and system prompts needed for every run. *Situational context* is just-in-time data loaded only when a specific scenario arises. He uses an analogy of a principal (who knows how classrooms should run) and a teacher (who has situational context about specific students). Loading a customer support ticket into context permanently would cause bloat; instead, the agent should pull it in via live lookup only when needed.

### Hack 1: CLAUDE.md as a Router [11:53](https://www.youtube.com/watch?v=Ek1NBfnnTH0&t=713s)
Nate's first hack is to treat the main `CLAUDE.md` file purely as a router rather than just a system prompt. His Herk 2 AIOS is one massive folder pushed to GitHub, containing subfolders for brainstorms, wikis, "other worlds" (standalone repos), and projects. The main `CLAUDE.md` acts as a master table of contents, directing the agent to exact paths for the wiki, hot cache, index, memory system, tools, API keys, and templates. This routing approach means the agent always knows where to look, regardless of how deep the folder structure gets.

### Hack 2: Have AI Audit Itself [16:35](https://www.youtube.com/watch?v=Ek1NBfnnTH0&t=995s)
Beyond the formal skill, Nate recommends simply having a conversation with your agent at the end of the week or month to review all files and routing rules. He shares that Claude Code itself suggested splitting his main wiki into two separate wikis (YouTube transcripts vs. meeting transcripts) to reduce bloat, confusion, and token costs. He emphasizes that there is no single "right" way to build a second brain—the only wrong way is to tolerate incorrect answers without iterating. A good test: if you can manually find a file in your explorer without searching, your agent can too.

### Hack 3: Build Automations to Update Data [19:07](https://www.youtube.com/watch?v=Ek1NBfnnTH0&t=1147s)
Nate stresses setting up crons to automatically pull in recurring data. For example, if he has a Q&A every Monday and a leadership meeting every Tuesday, he automates the ingestion of those transcripts so the data is always fresh in his local project. He notes this is simple to set up with natural language—just ask Claude to create the cron or push a script to a service like Modal.

### Hack 4: Segment Knowledge [20:09](https://www.youtube.com/watch?v=Ek1NBfnnTH0&t=1209s)
Building on the earlier point about splitting wikis, Nate explains that segmenting distinct, growing knowledge nodes makes searches faster and cheaper. For client work, he recommends keeping internal client knowledge (contracts, discovery calls, scope) inside the main AIOS, but building external deliverables in a separate, client-facing repo. This keeps the internal OS clean while still giving the agent enough context about the engagement.

### Hack 5: Backtrack [22:44](https://www.youtube.com/watch?v=Ek1NBfnnTH0&t=1364s)
When an agent fails to find data it should have access to, Nate doesn't just tell it to "try harder." Instead, he forces it to backtrack: retrace its search steps, identify where it looked and where it should have looked, and explain why it failed. Once the agent articulates its own mistake, Nate has it fix the routing rules or reorganize files to prevent the same failure. This proves far more effective than simply saying "don't let that happen again."

### Team-Level Syncing and Closing [23:46](https://www.youtube.com/watch?v=Ek1NBfnnTH0&t=1426s)
Nate acknowledges that syncing AIOS data across teams and departments is a hot topic, but he believes it's currently a people and habit problem rather than a technology problem. Tools like Google Drive, Notion, or GitHub can handle the syncing, but getting people to shift habits, know what to pull in, and manage permissions is the real challenge. He advises mastering your own personal AIOS first before attempting team-level implementation.
